## 1. Discovery (short)

- **Audience.** Worker JHSC reps, the worker co-chair, certified members. Ontario workplaces of 50+. Personal Android phones (often older, weak signal on the shop floor) and occasional personal laptops. Includes workers with disabilities — AODA is a hard constraint, not aspirational.
- **Primary task.** Capture inspection / concern / reprisal data quickly from a phone, and (for the co-chair) complete a deliberate friction-by-design export to the employer co-chair without leaking concern-derived or C4 content.
- **Content shape.** Forms-heavy (intake, inspection, recommendations) + list-heavy (concerns register, audit feed, sessions) + small set of high-consequence interstitials (export, panic-wipe, 4-eyes, lock screen).
- **Tone.** Plain, declarative, never coy. Says what the system can see, what it cannot, and what it logs. Uses the literal verb ("reprisal", "wipe this device's data", "export to employer co-chair") rather than euphemism.

## 2. Visual direction — committed

**Name:** **Civic-record.** A serious workplace-safety instrument that reads like a labour-tribunal binder rendered for a phone.

**Three references, with what we borrow from each:**
1. **GOV.UK Design System** — typographic posture, focus-ring discipline (visible black-on-yellow outline), unapologetic plainness of language.
2. **Stripe Dashboard** — calm semantic color (single accent + clearly-coded states), destructive confirm pattern that names the irreversible thing about to happen.
3. **Signal Desktop** — sensitivity vocabulary (C3 / C4 badge treatment), conspicuous lock affordance, "sealed" wording that tells the user what the system can and cannot see.

**Mood adjectives:** sober, audited, candid, careful, load-bearing.

**Three anti-patterns (forbidden):**
1. Gradients, glass-morphism, or marketing-grade hero illustrations on any sensitive surface (concerns, reprisal, work-refusal, s.51, exports, sessions). The product is not aspirational.
2. Icon-only controls without text labels for destructive, sensitive, or export actions. Icons accompany text; they do not replace it.
3. Color-only state signals. Sensitivity, success, error, warning, queued, and offline always pair color with an icon **and** a text label.

## 3. Interaction patterns (load-bearing across all surfaces)

### 3.1 Focus management
- **Focus-visible always.** Two-layer ring: `outer` (#fbbf24 / yellow halo, 3px) + `inner` (foreground line, 2px) per `shadow.focus_ring`. Visible on any background light or dark. **`outline: none` is forbidden without a replacement at least as visible.**
- **Skip-to-content link** at the top of every page; visually hidden until focused, then surfaces at top-left.
- **Tab order follows DOM order.** No `tabindex` other than `0` (or `-1` for programmatic focus targets).
- **Modal opens:** focus moves to the first focusable element inside the modal (or to a labelled close button), focus trap engages, body scroll locks.
- **Modal closes:** focus returns to the element that opened it.
- **Toast appears:** focus does NOT move automatically (interrupting screen-reader flow is hostile); toast is announced via `aria-live` (`status` for info/success/warning; `alert` for danger and sensitive).

### 3.2 Modal / Dialog rules
- One modal at a time. A second `show()` call closes the first.
- `Escape` dismisses **except** for `destructive_confirm`, `export_interstitial`, `reauth_prompt`, `passphrase_prompt`, `four_eyes_pending` — these require explicit Cancel (a coerced user must not be able to accidentally swipe a sensitive action away).
- Click-outside dismisses **only** for the plain `dialog` variant. Forbidden for the same five high-consequence variants above.
- Scrim color: `color.background.scrim`. z-index: `z_index.scrim` for the dimmer, `z_index.modal` for the dialog.

### 3.3 Toast / Notification rules
- **Severity → role:** info/success/warning use `role="status"` (polite); danger and sensitive_activity use `role="alert"` (assertive).
- **Auto-dismiss:** info/success 4s; warning 7s; danger and sensitive_activity **never auto-dismiss** (manual close only).
- **Max 3 visible** at once; oldest fades when a fourth arrives.
- **Sensitive-activity toasts** (RA-1 post-export notification; C4 sensitive-read notification) additionally write to the persistent audit feed, so a missed toast is recoverable.

### 3.4 Form validation timing
- **Inline validation on blur** for individual fields (so a sighted, sequential filler sees errors as they finish a field).
- **Batch validation on submit** for the form as a whole (so a screen-reader user filling top-down isn't yelled at mid-field).
- **Passphrase and TOTP fields validate only on submit.** Mid-typing validation of secrets leaks correctness through timing and through visible feedback.
- **Error messages** name what's wrong, why, and what to do. Never "Invalid." or "Error 422."

### 3.5 Destructive-action confirmation pattern (used by export, panic-wipe, 4-eyes, soft-delete on C4)
A single canonical pattern; do not reinvent.

1. **Trigger.** Destructive button (variant: `destructive`) — labeled with the literal verb ("Wipe this device's data", "Export to employer co-chair", "Delete reprisal entry").
2. **Modal variant: `destructive_confirm`.** Scrim + dialog. Headline = the action in plain English. Body = (a) what will happen, (b) what is irreversible about it, (c) any sensitive cargo it carries (e.g., "this export includes 2 concern-derived items").
3. **Inputs.** Most destructive_confirm modals require the user to either (a) re-auth via passkey, (b) enter a per-record passphrase, OR (c) type a literal confirmation phrase ("WIPE" for panic-wipe). The choice depends on the surface (see component spec).
4. **Buttons.** Cancel (variant `secondary`, on the left). Confirm (variant `destructive`, on the right). The confirm button is disabled until the requirement above is satisfied.
5. **Post-confirm.** Loading state in the modal (spinner + "Working…"); success closes modal and shows a confirmation toast; error stays in the modal with the error message inline.
6. **Audit-log emission.** Per RA-1 and Invariant 8: the audit row is the precondition, not a side effect — the action does not complete until the audit row is hash-chained.

**Specific behavior notes for A:**
- Auto-dismiss is **forbidden** for this modal in all states. Friction is the point.
- The recipient destination is rendered in plaintext as part of `reviewing-fields`: `export.recipient.label` = "Recipient: Employer co-chair (manual delivery — you will send the file yourself)." This makes RA-1 compensating control #1 visible.
- Per F-25 / F-24: no server-side PDF rendering, no POST that returns `application/pdf`. The implementer assembles bytes in the browser only.
- Post-export rep notification (RA-1 compensating control #4) is fired by the same submit action that creates the audit row; if the notification POST fails, the export still completes (audit row is the gate) but a `toast.warning` informs the co-chair that other reps will be notified on next sync.

**Accessibility packet (AODA / WCAG 2.0 AA — the mandatory handoff).**

- **List vs. table semantics.** The roster ships as **list semantics** (`<ul>`/`<li>`), mirroring `/concerns` and `/reprisal` for cross-surface consistency and mobile-first stacked rows — the roster is read-only in P1-8b (no cell-by-cell comparison need). Each `<li>` KEEPS its implicit `listitem` role; a **wrapper `<div>` INSIDE the `<li>`** carries `role="group"` + `aria-labelledby` pointing to its name element, so a screen reader announces each member as a self-contained unit (name → badge → roles → contact → date, in DOM order) without breaking the list structure. (Axe-clean resolution decided at P1-8b build time: `role="group"` directly on the `<li>` strips the implicit `listitem` role and fails axe's `list` rule, so the group role lives on an inner wrapper `<div>` — either satisfies "announced as a named unit" while keeping the `<ul>`/`<li>` valid.) The `<ul>` carries `aria-label="Committee members"` (`committee.roster.list_aria`). **If** a future desktop-columned variant is wanted (Surface H/I precedent), it MUST use a real `<table>` with `<caption>`, `<th scope="col">` per column, and `role` mapping — but that is out of P1-8b scope; the stacked list is the P1-8b deliverable.
- **Badge programmatic label — never color-only.** Each badge is `<span class="badge badge-*">` = decorative icon (`aria-hidden="true"`) + visible micro-text that already carries the meaning ("Active", "Waiting for key access", "Removed", …). Additionally attach the longer `committee.roster.badge.*.sr` string via `aria-label` on the badge (or a visually-hidden `<span>`), so an SR user hears the full, unambiguous status ("Waiting for you to grant committee-key access") rather than the compact visible label. Satisfies anti-pattern #3 (color paired with icon AND text) AND the §6.7 color-blindness check.
- **Not-a-co-chair announced POLITELY.** The stop state is `role="status"` (polite), never `role="alert"` — it is an EXPECTED authorization boundary, not a failure. A member who lands here (including a coerced or merely curious one) must not be alarmed. Contrast with the session-expired and generic-error states, which ARE `role="alert"` (assertive — a real interruption requiring action).
- **Loading via `aria-busy` + live region.** While `listRoster()` is in flight, the roster container carries `aria-busy="true"` and a `role="status"` live region announces the literal action ("Loading the committee roster…") — never bare "Loading…". On success, `a11y.committee.roster.loaded` announces the member count.
- **Focus order.** DOM order, no positive `tabindex`. Interactive elements in P1-8b are limited to: the `/more` nav link (to reach the route), the sign-in link (signed-out / session-expired states), the "Back to more" link (not-co-chair / list states), and the Retry button (error state). The roster rows themselves are NON-interactive (not tab stops) in P1-8b; the two-layer AODA focus ring (`shadow.focus_ring` via app.css `:focus-visible`) applies to every link/button. When screens 3/5 add per-row grant/role controls, those controls join the tab order and MUST carry the ring + ≥44px targets.
- **Keyboard operability.** No custom key handling — every interactive element is a native `<a>` or `<button>` (Tab to focus; Enter on links; Enter/Space on buttons). Escape is not consumed (no modal on the roster surface in P1-8b).
- **Nav entry accessible name.** The `/more` `.more-link` for /committee has accessible name = its `<strong>` label + `<span>` blurb text (`committee.nav.label` + `committee.nav.blurb`); any icon added is `aria-hidden` with the text label present (anti-pattern #2 — never icon-only for a navigational/consequential control). `aria-current="page"` is NOT set on the /more row (the destination is /committee, not /more).
- **Touch targets.** The `/more` nav row is a full-width padded link (well ≥ `touch_target.min` 44px). The Retry / Back / sign-in controls are `button`/`.cta`/link at `min-height: 2.75rem` (44px) per app.css. Rows are non-interactive so carry no target requirement in P1-8b; later per-row action targets MUST be ≥44px (`touch_target.min`, general app tier — the roster is not in the gloves-friendly inspection/reprisal 48px tier).
- **Reduced motion.** The only motion is the optional loading spinner; under `prefers-reduced-motion: reduce` it degrades to static text + the `aria-busy` live region per §6.6 (no rotation). No row-insert animation (the roster is a single fetch, not a stream).

**The single load-bearing control (F-170 — read this first).** The 6-digit code is the OUT-OF-BAND secret; the redeem link is NOT secret (it carries only `invite_id`). The design MUST keep those two custodies split so that intercepting one channel never yields both:

- **The ONLY clipboard affordance copies the LINK.** There is **no** "copy code" button, **no** "copy both", **no** "share invite" combined affordance anywhere in screens 2 or 4. The one copy button is the reused `ShareUrlButton`-shape control bound to `buildRedeemLink({invite_id})` — it copies the link string and nothing else.
- **The code is displayed as selectable static text, never in an `<input>`, never with a copy/share icon.** The co-chair conveys it by reading it aloud or typing it into a *different* channel than the link. OS-level text-selection of the code is unavoidable and acceptable (the co-chair has to convey it somehow); what the app must NOT do is hand them a one-tap "copy the code" that makes copying the code *and* the link into one gesture the easy path.
- **The code is shown ONCE and is not recoverable.** It exists only in an in-memory variable on the co-chair's device (F-176 — never logged, never in `sessionStorage`/`localStorage`/IndexedDB, never appended to any URL, never echoed by the server). Dismissing the panel discards it; if lost, the recovery is a fresh **re-send** (screen 4), which mints a NEW code and invalidates the old one.
- **No auto-send / no delivery surface.** No email field, no SMS, no "notify member" toggle, no server-side send. Out-of-band delivery by the co-chair is the deliberate zero-subprocessor compliance win (F-170 ruling); the UI provides only the copyable link + the readable code + the guidance to split them.

A design that lets the co-chair copy the code and the link together in one gesture **defeats F-170** and is a blocker, not a polish note.

**Entry affordance (recommended: inline panel, mirroring `/concerns`).** On `/committee`, at the top of the roster section, add an **"Invite a member"** primary CTA (`button.primary`, ≥`touch_target.min`). Tapping it toggles-to-mount an inline invite **panel** (`card.default`) directly below the CTA — the same toggle-to-mount create pattern `/concerns` uses for "Log a concern" (`concerns/+page.svelte:74-76`; each open is a fresh component instance, so the code/role state never leaks across invites). **Recommended over a modal** because (a) it matches the app's established create pattern (no modal-dialog primitive exists in the app for create flows — modals are reserved for the destructive/interstitial tier: `ExportInterstitial`, `PanicWipeModal`), and (b) the F-170 "shown once, not recoverable" property is *safer* in a panel that clears only on an explicit Done than in a modal whose click-outside / Escape could swipe the once-shown code away by accident. **Modal fallback contract (if the implementer chooses a modal anyway):** the `code_shown` state MUST be treated as a **protected variant** per §3.2 — click-outside and Escape are FORBIDDEN while the code is displayed (join the five protected variants); dismissal is the explicit Done button only. The `form` state may be a plain `dialog` variant (Escape/click-outside allowed) since nothing irreversible has happened yet.

**Shared building block — the one-time-code custody card (rendered by BOTH screen 2 `code_shown` and screen 4 `code_shown`).** Specify once; both screens compose it. Top-to-bottom:

1. **Panel heading** (`typography.roles.subsection`, `color.foreground.primary`) — screen 2: `committee.invite.code.heading` ("Invite created — share the code out of band"); screen 4: `committee.resend.code.heading` ("New code ready — the old one has stopped working"). Focus moves here on transition (see a11y packet).
2. **The 6-digit code**, in a **high-contrast display box** (NOT an input): box bg `color.background.primary`, border `border_width.thick` (2px) `color.border.strong`, `radius.default`, padding `spacing.4`; the digits in `typography.roles.totp` (large tabular mono), `color.foreground.primary` — the same max-contrast foreground/background pair Surface D.T19.f uses for the recovery reveal (16.1:1 light / 15.2:1 dark, contrast-audited). Rendered as a `<code>`/`<span>` of selectable text; **no `<input>`** (paste-history / autocomplete leak, mirroring D.T19.f), **no copy icon, no share icon**. Above the box, a micro-label `committee.invite.code.label` ("One-time code") in `typography.roles.label_caps`.
3. **Custody-split callout** — the load-bearing security instruction, `alert_banner.warning` (`color.state.warning_bg`/`warning_border`, 4px left border) with **alert-triangle icon + text** (color never alone): `committee.invite.custody.heading` ("Send the code and the link separately") + body `committee.invite.custody.body` ("Read the code out loud, or send it in a different app than the link. Never put the code and the link in the same message — if someone sees both, they can take over this invite."). This is where the F-170 mental model is taught to the co-chair.
4. **The redeem link** — micro-label `committee.invite.link.label` ("Redeem link (safe to send)"); the URL string `buildRedeemLink({invite_id})` in `typography.roles.code` (mono) inside a `color.background.secondary` box; a **"Copy link"** button (the reused `ShareUrlButton` shape: `button.secondary`/`.btn-outline`, `aria-live="polite"` "Link copied", transient `color.state.success_*` copied state / `color.state.warning_*` failure state — verbatim the existing component's behaviour). Helper `committee.invite.link.helper` ("This link only points to the invite. It does not contain the code, so it's safe to email or message.").
5. **"Shown once" reminder** — `alert_banner.info` (neutral) with info icon + text: `committee.invite.code.once` ("This code is shown once. If you lose it before the member joins, send a new one from Pending invites.").
6. **Dismiss** — `committee.invite.code.done` ("Done — I've shared the code"), `button.primary`, ≥`touch_target.min`. On click: clears the in-memory code, closes the panel, returns focus to the entry CTA (screen 2) or the originating pending-invite row's Re-send button (screen 4). Secondary `committee.invite.code.resend_now` ("Send a different code") — `button.ghost` — re-runs the mint for the *same* invite (screen 4 semantics, available in both screens so a co-chair who fumbled the read-aloud can re-mint immediately) and replaces the displayed code; the old one dies.

The response fields `invitee_user_id` and `bootstrap_id` are **NEVER rendered** (F-176 — `bootstrap_id` is TOTP-secret-adjacent; the raw invitee uid is not needed by the co-chair at invite time). Only `invite_id` (→ the link) and the CLIENT-held code reach the UI.

**Screen 4 — Re-send code — where it lives + reconciliation.** A **distinct section titled "Pending invites"** renders **below** the roster inside the same `/committee` `<main>` (its own `card.default`, `typography.roles.section` heading `committee.resend.section.heading`). **Recommended: a stacked section, NOT a tab** — the app has no tab primitive for content panes (`BottomTabBar` is the app nav, not a content tabset), and the mobile-first single-scroll pattern (`/concerns`, `/reprisal`) argues for a section the co-chair scrolls to. It is backed by **B2 `listPendingInvites()`**, called on `/committee` mount alongside `listRoster()`.

**Roster (B1) vs. Pending invites (B2) — the reconciliation (pinning the orchestrator's flagged ambiguity).** These are **two distinct sections that intentionally overlap by design, NOT a duplication bug:**

- The **roster (B1, screen 1)** is the authoritative *membership* list. An invited-but-not-redeemed member already appears there as a **read-only `pending-invite` badge row** (`active=false, activated_at=null`) — it answers *"who is on the committee, and what's each member's state?"*
- The **Pending invites section (B2, screen 4)** is the *actionable invite-management* projection (`committee_invite` rows where `consumed_at IS NULL`, carrying `invite_id` + `expires_at`). It answers *"which invites are outstanding, and which need a fresh code?"* — and it is the ONLY place the per-invite **Re-send** action lives.
- **So a pending member legitimately appears in BOTH** — once as roster signage (no action), once as an actionable invite. This is acceptable because the two sections answer different questions and are visually + semantically separated (distinct headings, distinct cards). To keep it from reading as an accident: the roster's `pending-invite` badge row carries a cross-reference hint `committee.roster.badge.pending_invite.hint` ("Manage in Pending invites") in `typography.roles.small` / `color.foreground.muted`, and the Pending-invites section heading has a one-line blurb `committee.resend.section.blurb` ("Invites that haven't been redeemed yet. Re-send a fresh code, or invite again if one has expired."). **Do NOT interleave B2 rows into the B1 roster** (that would double-render inside a single list). Confirm this two-section split is intended — see Open questions.

- **The code is announced, not focus-stolen.** On transition to either `code_shown`, focus moves **once** to the custody-card heading (`tabindex="-1"`) — a single deliberate move mirroring Surface J's success pattern, NOT a repeating assertive live-region that would talk over a co-chair reading the code aloud. **The load-bearing property — read this first (F-180, and it is the OPPOSITE of a naive reading).** This screen renders the member's identity fingerprint so the co-chair can compare it out-of-band against what the member reads from their P1-9 waiting screen (Surface L) before sealing the committee data key to that member's pubkey. **That human compare is an ADVISORY, defense-in-depth check — it is NOT the crypto gate, and the copy MUST NOT imply that it is.** The load-bearing controls are all server-bound (F-180 `:4028`): (1) the seal target is the SERVER-disclosed pubkey from `getMemberPubkey` — the co-chair client cannot fabricate or substitute one, and the member's *rendered* fingerprint never feeds the seal; (2) the A-8.6 single-disclosure self-consistency assert — `wrapMemberInViaProduction` re-derives `pubkeyFingerprint(disclosed.public_key)` and typed-fails unless it equals the confirmed `disclosed.fingerprint`, so the confirmed bytes ARE the sealed bytes (TOCTOU closed); (3) the active-member re-assert at seal time; (4) `auth.uid()`-scoped wrap delivery (a wrong seal is target-DoS, never exfiltration). What the human compare buys: it catches an **honest co-chair about to grant to the wrong member row**, and a member-side display tamper the two humans would notice. A co-chair who *skips* the read-aloud degrades to the server controls above — **no hole opens.** Two consequences bind this design: **(a)** the copy frames the compare as "an extra check that you're granting to the right person," never as "this is what keeps the key secure"; **(b)** the `granted` and `failed` terminal states render **ONLY** from the server's `WrapMemberInResult` return — the UI NEVER optimistically implies a success the server did not confirm (the F-181 discipline applied to F-180).

**The advisory-not-load-bearing framing — the exact copy discipline (F-180).** The compare callout and the confirm CTA carry the whole F-180 framing; every word is chosen so the co-chair understands the compare as a *human double-check layered over server controls*, never as the thing that secures the key:

- `committee.grant.compare.body` → "Ask {name} to read their fingerprint from their own device, one group at a time — or have them show you their screen. Check every group matches the one above. This is an extra check that you're granting access to the right person; the app already makes sure the key only goes to the member who set up encryption." — the final clause names the server as the control (per F-180 enforceable #1), so the read-aloud is positioned as *belt-and-suspenders*, not *the belt*.
- `committee.grant.confirm.cta` → "It matches — grant access" — the button label makes the co-chair's affirmative claim explicit (so tapping is a deliberate "I checked"), WITHOUT a separate gate that would mislabel the advisory step as load-bearing.
- **No copy anywhere on this screen may say** the read-aloud/visual compare "verifies", "secures", "protects", or "confirms the identity of" the member in a way that implies the seal depends on it. The seal depends on the server-disclosed pubkey + the single-disclosure self-consistency assert (A-8.6), full stop.
- **Terminal states are server-truthed.** `granted` renders ONLY on `{ status:'ok' }`; `not_ready` ONLY on a `member_not_enrolled`-class return; `failed` ONLY on a `failed`-class return. The UI never flips to `granted`, and never drops/greys the roster row, before the wrap returns success (the F-181 anti-optimism discipline, applied to F-180).

**⚠ DIVERGENCE — read this before anything else. The committee 4-eyes is SINGLE-CALL INLINE, NOT the app-wide two-phase ceremony.** §3.2 lists `four_eyes_pending` among the protected modal variants and Surface C (`design-system.md:155-157`) specifies a generic **two-phase** 4-eyes (`4-eyes-pending` / `4-eyes-confirmed`): a co-chair *proposes* a change, it is *recorded* with a TTL, and a **distinct second member approves it later, asynchronously**, in their own feed. **Screen 5 is NOT that ceremony and the implementer MUST NOT build the two-phase flow here.** The committee RPCs (`committee_set_roles` / `committee_remove_member`) take an **inline `p_second_approver_id`** argument on a **single call**: the acting co-chair names a second co-chair *in the same request*, and the server accepts iff that id is a **distinct active worker co-chair** (`_committee_is_active_co_chair`, `sql:346-350,405-410`). There is **no proposal record, no TTL, no pending state, no async second-party approval step, no `four_eyes_pending` waiting screen** on this surface. The second co-chair does not act in-app; the acting co-chair designates them (having obtained their concurrence out-of-band — the honest framing the copy uses, below). Reprisal-status-change (Surface C) keeps its two-phase flow; committee membership governance uses this single-call inline form. Do not cross-wire them.

**The load-bearing property — read this second (F-181 anti-optimism + F-182 honest-limit + the self-action-only 4-eyes trigger).** Three hard postures govern every word and every state:

1. **The 4-eyes trigger is SELF-ACTION-ONLY, and the server is the sole truth (F-181).** The SQL requires a second approver in **exactly one shape**: the acting co-chair acting on **themselves** — self-demoting *out of* worker-co-chair (`committee_set_roles` `v_losing && v_actor = p_target_user_id`, `sql:336-351`) or self-removing while a worker co-chair (`committee_remove_member` `'worker_co_chair'=ANY(role) && v_actor = p_target_user_id`, `sql:396-411`) — **and only when they are not the last co-chair** (the `last_co_chair` guard is checked FIRST, `sql:342-345,401-404`). Acting on **another** member — co-chair or not — needs **no** second approver (the `IF v_actor = p_target_user_id` branch is simply not entered); only the `last_co_chair` guard can bite an other-co-chair target, and because the acting co-chair is itself always an active co-chair, an other-co-chair action can never drop the count to ≤1, so in practice **`last_co_chair` and `4eyes_required` arise ONLY on a self-action.** Terminal `done` renders **ONLY** from a clean RPC return (void / the grace timestamp — no raised reason). A `4eyes_required` return → the `4eyes_required` state (NOT done). A `last_co_chair` return → the `last_co_chair` blocked state (NOT done), surfaced as an **explained block keyed off the server reason, never off a client-side count**. Reasons are **discriminated** (`4eyes_required` vs `last_co_chair` vs `invalid_role` vs `not_found` vs `already_active` vs `rls_denied`), never collapsed to a generic error. The row badge NEVER flips before the server confirms (server re-fetch on `done`, never an optimistic client flip — the same discipline as screen 3's `granted`).

2. **The second-approver picker matches the server EXACTLY (never demand one the server would waive; never dead-end one the server requires).** The picker appears **precisely when the server will require it**: a **self-action that drops the actor's own worker-co-chair role**, when at least one **other** active worker co-chair exists. Its options are the **eligible-approver list derived client-side from the already-loaded roster** `rows` + `getCurrentUserId()`: `rows.filter(r => r.active && r.roles.includes('worker_co_chair') && r.user_id !== getCurrentUserId())`. This mirrors `_committee_is_active_co_chair` (**`active && 'worker_co_chair' = ANY(role)`**) minus self — keyed off the **`active` boolean + role membership, NOT the `active` badge** (a co-chair who is `pending_grant`/`awaiting_identity` — `active===true` but no live wrap / no identity key — is STILL a valid `second_approver_id` per the SQL, so the picker MUST include them, or it would falsely block an action the server allows). Confirm stays **disabled until a distinct eligible approver is selected** — "distinct" is guaranteed by construction (the options exclude the acting co-chair, who in a self-action is also the target, so the pick is `!= v_actor` and `!= target`). When the eligible list is **empty** (the actor is the only active worker co-chair), the UI shows **no picker** and does **not** gate Confirm on it (demanding an approver that cannot exist would dead-end self-removal, which is forbidden) — it shows an advisory heads-up and lets the co-chair Confirm, then renders the **server-truthed `last_co_chair` block** (empty eligible list ⟺ actor is the last co-chair ⟺ the server raises `last_co_chair`; the block is still rendered from the reason, not the count). For **other-member** actions the picker is **not offered at all** (decision below).

3. **Removal is NON-cryptographic and the copy MUST be honest (F-182 — the load-bearing copy of this screen).** `committee_remove_member` sets `active=false`, `deactivated_at=now()`, `grace_until = now()+90 days`, and nulls `public.users.role` (`sql:414-417`) — it does **NOT** rotate the committee key and does **NOT** delete the removed member's device wrap. A removed member who kept their device could still decrypt committee data until a key-rotation ships (a post-Phase-1 follow-up). The remove-confirm copy therefore states the limit honestly ("removes their committee membership", "does not rotate the shared committee key or re-encrypt existing records", "an administrative step, not a cryptographic lockout") and contains **NO** claim that removal "revokes access to committee data" / "they can no longer decrypt" / "revokes data access" or any equivalent (an F-182 string-test forbids these). Symmetrically, `committee_reactivate_member` restores `active=true` and re-derives `users.role` from the **retained** `committee_membership.role` (`sql:453-457`) — it does **NOT** run a fresh grant. The reactivate-confirm copy states access returns via the member's **retained wrap**, not a fresh grant ceremony.

**Where screen 5 lives + the attach pattern (mirror the P1-8d grant card — optional deps, actions only when wired).** Screen 5 is **not** a route. It adds per-row management affordances inside the existing `/committee` roster (app-shell layout 5.1), mirroring `CommitteeRoster.svelte:436-443` verbatim in shape: the route threads optional management deps into `CommitteeRoster`; when they are wired a per-row child (`CommitteeManageMemberCard.svelte`, the screen-5 sibling of `CommitteeGrantCard`) mounts **inside the row's `role="group"` wrapper** (so the SR reads the controls as part of that member's unit); when they are absent (the P1-8b read-only + P1-8d grant-only compositions) **no management control renders on any row** and the roster stays read-only. The parent (`CommitteeRoster`) computes **once** — `currentUserId = getCurrentUserId()` and `eligibleApprovers = rows.filter(active && roles.includes('worker_co_chair') && !== currentUserId)` — and passes to each card `{ member: RosterRow, isSelf: member.user_id === currentUserId, eligibleApprovers, client }`. The card owns the modal state machine and, on any successful mutation, emits `on:changed` → the parent re-runs `listRoster()` (server-truthed badge refresh; **no optimistic flip** — F-181). Recommended enablement prop: a single `export let manageEnabled = false` on `CommitteeRoster`, set true only by the `/committee` route (which already renders the roster only for co-chairs); the card additionally no-ops if `client` is the inert default.

> **Status:** Designer pass 2026-07-13 — **this pass scopes ONLY the new `waiting` phase (and its `computing` → `shown` → `derive_error` sub-states) inside the existing `SetupCommitteeEncryptionCard.svelte` on `/settings`.** No new route, no new component. **The load-bearing property (F-172 — read this first).** This phase renders a member their OWN identity fingerprint while they wait for a co-chair to grant committee-key access. During the P1-8d grant, the co-chair calls `getMemberPubkey`, derives the **same** fingerprint via the **same** `pubkeyFingerprint()`, and compares it out-of-band against what the member reads aloud / shows on screen before running `wrapMemberInViaProduction`. **That human comparison is an advisory, defense-in-depth check layered over the server-side controls — it is not, on its own, the crypto boundary.** The load-bearing guarantees are server-bound: the seal target is the server-disclosed pubkey, and the A-8.6 single-disclosure self-consistency assert makes the confirmed bytes the sealed bytes — so a co-chair who skips or mis-reads the compare degrades to those controls and no hole opens. What the compare still buys is real: it catches an honest co-chair about to grant to the wrong member, and a member-side display tamper the two humans would notice — so an accurate, group-for-group read-aloud is still worth getting right. So the entire treatment (grouping, wrapping, and above all the screen-reader read-aloud technique) is optimized for *accurate human conveyance of a 64-character string*, not for visual density. 64 hex is a far harder conveyance problem than the 6-digit invite code (Surface K); the grouping + the SR announcement format below are the mitigations.

**Copy-of-the-fingerprint is ALLOWED (deliberate contrast with Surface K).** The fingerprint is a **public** value — it is `SHA-256(public_key)`, derived from the identity *public* key, and the co-chair independently re-derives it. It is NOT a secret. Therefore, unlike the Surface K one-time code (where copy-of-the-code is **forbidden** because the code is the out-of-band secret whose custody must stay split from the link), here a **"Copy fingerprint"** affordance is correct and helpful: it lets the member paste the canonical string into whatever channel the co-chair is comparing against, and lets the co-chair paste-compare instead of eyeballing 64 chars. The F-176 attribute-sweep constraint that applies to the invite code does **not** apply here — the contiguous fingerprint may appear in the copy button's `url` prop, the clipboard, and the a11y tree freely.

**Entry gate (where it sits in `probeState()`).** The gate is caught **before** the final `not_provisioned` fall-through at `:205-209`. At that point in the machine, every `!hasDevicePrivkey` branch has already returned (`:174` restore_required, `:183`/`:194` foreign_held), and the `actor_has_wrap && hasDevicePrivkey` branch returned to `hidden` (`:168`), so **`hasDevicePrivkey === true && state.actor_has_wrap === false` is guaranteed** and the only remaining discriminator is `wrap_count`:

- `state.wrap_count > 0` → **`waiting`** — a committee key exists, is held by other members, and this actor has done their own part (identity enrolled + privkey on this device) but has no wrap yet. They are waiting for a co-chair to grant. **This is the new branch.**
- `state.wrap_count === 0` → **`not_provisioned`** (unchanged) — no committee key exists at all; this member is the first co-chair and initializes it.

So the gate is exactly the orchestrator's condition: `hasDevicePrivkey === true && state.actor_has_wrap === false && state.wrap_count > 0`. **The fingerprint is derived ONLY in this branch**, where `identityPrivkey` (set at `:146`) is guaranteed to be a 32-byte key — the `derive_error` sub-state is a defense-in-depth fail-safe that should not occur here.

Add to the card:
- `phase` union gains `'waiting'`.
- a sub-state var `let waitingState = 'computing' | 'shown' | 'derive_error'` (mirrors the existing `restoreState` pattern).
- `let fingerprint = ''` (the contiguous 64-hex, canonical/lowercase — the copy value + the string groups are sliced from).
- a `deriveWaitingFingerprint()` function analogous to `resumeWithDeviceKey()` (`:236`): `const s = await ready(); const pub = s.crypto_scalarmult_base(identityPrivkey); fingerprint = await pubkeyFingerprint(pub);` set `waitingState='shown'` on success, `waitingState='derive_error'` on any throw. In `probeState`, the gate sets `phase='waiting'; waitingState='computing'; void deriveWaitingFingerprint(); return;` (fire-and-forget so the `computing` render is honoured, exactly how the card kicks off its other async steps).

**Grouping (pin this — it must match Surface K screen 3 byte-for-byte).** The 64-hex string is split into **16 groups of 4 characters** (`fingerprint.match(/.{4}/g)` → 16 items). Visual layout: groups flow left-to-right and **wrap only at group boundaries** — a 4-char group is **atomic** and MUST NOT split across a line (splitting a group destroys the read-aloud position contract). Recommended block shape: 4 rows × 4 groups on ≥`breakpoint.sm`, wrapping to fewer-groups-per-row on narrower phones; groups separated by a visible space (column gap `spacing.2`, row gap `spacing.1`). Mono, selectable, tabular. **Cross-surface invariant:** the P1-8d co-chair screen (Surface K screen 3, later pass) MUST use the identical 4-char/16-group split, same order, same lowercase, so the two humans compare group-for-group; record this so the screen-3 designer pass inherits it.

**Accessibility packet (AODA / WCAG 2.0 AA — mandatory handoff; the read-aloud technique is the load-bearing item).**

- **The core problem.** 64 hex read/heard once, over a phone or in person, and compared by a second human. The failure mode is a **skipped, duplicated, or transposed group** — not usually a single wrong glyph. So the SR technique must give (a) character-by-character reading within a group (no "four hundred eighty-two thousand…"), and (b) **stable positional landmarks** so a mis-sync is localizable and recoverable. A single monolithic `role="img"` over all 64 chars (the OneTimeCodeCard shape, correct for 6 digits) is **rejected here** — it is one un-navigable blob with no way to re-check one group or track position.
- **Recommended technique — grouped, numbered, spelled, navigable.** Render the fingerprint as an **ordered list**: `<ol role="group" aria-label="{a11y.settings.setup.fingerprint.region_label}">` of **16 `<li>`**, each `<li>` containing `<span role="img" aria-label="{group_label}">{group}</span>` where the visible `{group}` is the 4 hex glyphs (mono, selectable) and the `aria-label` is `a11y.settings.setup.fingerprint.group_label` = `"group 3 of 16, charlie 3 delta 4"` (a–f NATO-phoneticized per OQ-1 resolution below; the `role="img"` swaps the visible glyphs for this spelled label in the a11y tree — the exact `OneTimeCodeCard:145` mechanism, applied per group). Result: an SR user arrows group-by-group; each announcement carries its **own position ("group 3 of 16")** AND the **spelled chars ("c 3 d 4")**; the `<ol>`/`<li>` structure additionally gives "list, 16 items / 3 of 16" in item-navigation mode. This is re-checkable (arrow back to any group), position-aware (numbers are how the two humans re-sync), and forces digit-by-digit reading. Groups are **static, not tab stops** (16 tab stops would wreck the card's tab order) — they are reached via the SR reading/browse cursor.
- **A leading instruction for SR users.** Before the list, a visually-hidden (or visible) `a11y.settings.setup.fingerprint.ready` announcement (polite) tells the SR user the shape ("sixteen groups of four") so they know what's coming and how many groups to expect — a missed count at the end is then self-evident.
- **NATO-phonetic hex letters a–f, plain digits — RESOLVED (P1-9, accessibility-specialist).** OQ-1 is closed to a **hybrid**: the `aria-label` spells the six confusable hex **letters** `a–f` with NATO words (`alpha bravo charlie delta echo foxtrot`) and leaves **digits** `0–9` plain, so `c3d4` → `"group 3 of 16, charlie 3 delta 4"`. Rationale (WCAG 1.1.1 alt-text quality): spoken in isolation `b/c/d/e` collapse into the English "E-set" (bee/see/dee/ee), and this is the one channel where a single mis-hearing defeats F-172 — for a blind member the SR→ear→mouth→co-chair's-ear chain makes it worse. Digits are already unambiguous, so full-NATO was rejected (pure verbosity; no standard NATO digit words). The `compare_body` copy also coaches BOTH humans to "say each letter as a word" so a **sighted** member's read-aloud (which the SR label never touches) is protected too. Implemented in `waitingGroupLabel` (a–f → NATO map, digits pass through); no new token; the `group_label` template is unchanged (only its `{chars}` fill changed). **Screen-3 invariant:** the co-chair's P1-8d compare screen (Surface K screen 3) MUST mirror this exact hybrid so the two humans compare token-for-token. The threat-modeler co-signed that the glyph spelling has zero security bearing (identical public fingerprint either way).
- **Copy affordance a11y.** The **only** clipboard control is "Copy fingerprint" (reused `ShareUrlButton`, `fullTarget={true}` → `touch_target.min` 44px). It copies the **contiguous** 64-hex `fingerprint` (canonical lowercase, matching `pubkeyFingerprint()` output and the co-chair's paste-compare target). On success it announces `a11y.settings.setup.fingerprint.copied` ("Fingerprint copied to the clipboard.") via the component's built-in `visually-hidden aria-live="polite"` span **without moving focus**; on clipboard-unavailable it announces `..._copy_failed`. Text-labelled, never icon-only (anti-pattern #2). Unlike Surface K, a copy-of-the-value control here is correct (public value) — this is an explicit design contrast, not an oversight.
- **Contrast.** The fingerprint glyphs are `color.foreground.primary` on `color.background.primary` inside the bordered box — the app's audited max-contrast body pair (16.1:1 light / 15.2:1 dark, the same pairing D.T19.f / OneTimeCodeCard use for load-bearing secret/verify text). No new pair is introduced; comfortably AAA. The `border_width.thick` `color.border.strong` frame ensures the box is perceivable independent of the fill.
- **Focus on entry.** On `shown`, focus moves once to the lead paragraph (or the card `<h2>`) via `tabindex="-1"` (single deliberate move, mirrors Surface J) so a keyboard/SR user lands at the top of the new content and reads the lead → fingerprint → compare instruction in order. On `derive_error`, focus moves to the error heading (`role="alert"`).
- **Reduced motion.** There is no motion in this phase (no spinner in `computing`; no animated reveal) — nothing to degrade. The copy "Copied" swap is a text change, not an animation. Compliant by construction.
- **Color-blindness.** The info compare-callout and the error panel each pair tint with an icon AND text (anti-pattern #3); no state in this phase is carried by color alone.
- **Touch targets.** The Copy button (`fullTarget`) and the derive_error Try-again button are ≥`touch_target.min` (44px); adjacent-target spacing ≥`touch_target.spacing_between_targets_min`.

## 6. Accessibility handoff prep

(This section is the packet the accessibility-specialist consumes.)

### 6.1 Targets

- **Bottom nav** (`z_index.bottom_nav`, height `layout.app_chrome.bottom_nav_height` = 3.75rem, bg `color.light.surface.raised`, border-top 1px `color.light.border.subtle`):
  - 4 tabs, each min-target `touch_target.inspection_min`, icon (`icon.size.default`) + text label (`typography.roles.caption`).

**Every value is from a token. No invented values.** If the implementer cannot build this from `design-tokens.json` plus this spec, file a designer-pass-needed issue rather than inventing.
