## 1. System overview and trust boundaries

The worker-side JHSC app is a single-tenant, single-committee, worker-only PWA supporting the worker side of an Ontario Joint Health and Safety Committee (OHSA s.8–9). The adversary model is unusual and load-bearing: the **employer is treated as a credible, well-resourced adversary** alongside the conventional set of external attackers, insider threats, and platform compromises. The system must remain unreadable to:

- **A1 — Employer via Canadian or US legal process** (subpoena, production order, CLOUD Act / NSL / FISA 702 reaching Supabase Inc. or AWS).
- **A2 — Employer via a worker's employer-owned device** (MDM, screen-recording, key-logging, certificate interception on employer networks).
- **A3 — Co-opted or coerced insider rep** ("under duress" rep handing data to the employer in exchange for favour or under threat of OHSA s.50 reprisal).
- **A4 — Conventional external attacker** (phishing, session theft, dependency compromise, supply-chain).
- **A5 — Hosting provider compromise / rogue admin** (Supabase Inc. employee or compromised infrastructure).

The hosting decision (ADR-0001) is only defensible because the E2EE model (ADR-0003) holds. **Every STRIDE finding below is graded with reference to whether the E2EE boundary still confines plaintext to the user's browser.**

### Trust boundaries (architect's three, ratified, plus two surfaced here)

1. **B1 — Auth boundary (Supabase Auth perimeter).** Unauthenticated requests → authenticated principal. WebAuthn + short-TTL sessions. Outside B1: zero implicit trust. Inside B1: a named `auth.uid()` that RLS uses.
2. **B2 — E2EE boundary (client / server data plane).** Plaintext lives ONLY in the browser (libsodium buffers, Svelte component state, transient IndexedDB session-wrapping layer). Anything crossing toward the server MUST be ciphertext for C3/C4 data. Edge Functions are server-side and never see plaintext.
3. **B3 — Worker/employer boundary (the export function).** The single egress path off the worker side. PDF rendered in-browser by the worker co-chair, downloaded, delivered off-app. Logged in audit log.
4. **B4 — Browser session boundary (added here).** The architect's diagram conflates the IndexedDB-on-disk state with browser RAM. This threat model splits them: the contents of IndexedDB are at rest on a personal device and survive crash/reboot; the in-memory plaintext does not. Different threat surfaces (T2 employer-owned device hits IndexedDB; T6 device theft hits both). The implementer treats IndexedDB as a write-through ciphertext store with a session-scoped wrapping layer; raw plaintext is not allowed to persist.
5. **B5 — Backup/restore boundary (added here).** The nightly `pg_dump` crosses from the live Supabase project into an independent Canadian B2 bucket. The architect treats this as a ciphertext blob (correct), but the *dump encryption key* is its own escrow asset (paper + 1Password) — its compromise yields the same metadata the platform sees, plus the C1 audit log content. Worth its own trust boundary because the controls (key escrow procedure) are different.

## 2. Data flow diagrams (per priority component)

### 2.1 E2EE key handling (T07, ADR-0003)

```
[First-login enrollment]
  Browser: libsodium.crypto_box_keypair() -> {ident_pub, ident_priv}
  Browser: passkey-derived secret K_p = WebAuthn PRF / passkey-wrap (T05 + T07)
  Browser: ident_priv_wrapped_local = secretbox(ident_priv, K_p)
  Browser -> IndexedDB: store ident_priv_wrapped_local      [B4]
  Browser: prompt user to print recovery passphrase R; user types it back to confirm
  Browser: K_r = argon2id(R, salt)
  Browser: ident_priv_recovery_blob = secretbox(ident_priv, K_r)
  Browser -> POST /users/me/recovery_blob { ident_priv_recovery_blob, salt }   [B2, ciphertext]
  Server: stores in users.identity_privkey_recovery_blob (C2)

[Committee data-key generation — done once by first co-chair]
  Browser: libsodium.crypto_box_keypair() -> {ck_pub, ck_priv}
  Browser: wrap_for_self = crypto_box_seal(ck_priv, self.ident_pub)
  Browser -> POST /committee_key  { ck_pub, wrap_for_self }   [B2]
  Server: committee_key.public = ck_pub (C1); committee_key.wrapped[self] = wrap (C3)

[Add member (existing member M re-wraps for new member N)]
  M browser: unseal own wrapped ck_priv = crypto_box_seal_open(self.wrap, self.ident_priv)
  M browser: fetch N.ident_pub (server)
  M browser: wrap_for_N = crypto_box_seal(ck_priv, N.ident_pub)
  M browser -> POST /committee_key/wrap  { member_id: N, wrap: wrap_for_N }   [B2]

[Remove member X (rotation)]
  Remaining member R browser: takes per-committee advisory lock via Edge Function
  R browser: unseal own ck_priv (old key)
  R browser: generate new committee keypair {ck_pub', ck_priv'}
  R browser: for each remaining member Y including self: wrap_for_Y = crypto_box_seal(ck_priv', Y.ident_pub)
  R browser -> POST /committee_key/rotate { new_pub: ck_pub', new_wraps: [...] }   [B2, atomic]
  Server: in single transaction: (a) update committee_key.public; (b) replace all wraps with new wraps;
          (c) delete X's wrap; (d) audit log; (e) release lock.
  Server: NEW C3/C4 writes are sealed to ck_pub'. OLD ciphertext (sealed to ck_pub) remains; remaining
          members can still decrypt via their OLD wrap (kept under a `committee_key_history` row keyed
          on rotation_id) — X's old wrap is destroyed so X cannot.

[Submission of a concern body (sealed-box submission)]
  Browser: ct = crypto_box_seal({title, body, source_name?}, ck_pub)
  Browser -> POST /concerns { ct, metadata }   [B2]
  Server: row in concerns: ct columns + metadata; audit log row.

[Decrypt path]
  Browser: fetch self.wrap (current) and the ciphertext
  Browser: ck_priv = crypto_box_seal_open(self.wrap, self.ident_priv)
  Browser: plaintext = crypto_box_seal_open(ct, ck_pub, ck_priv)
  (For C4: an additional layer with a per-record key wrapped under ck_pub; a second decrypt step.)
```

### 2.2 Concern intake (T08, ADR-0007)

```
[Auth gate] Rep is authenticated; RLS evaluates is_active_member() = true.
[Form] Rep opens intake; anonymous toggle defaults ON. The source-name input is hidden behind the toggle.
[Client encrypt] Title + body + (optional) source_name encrypted with ck_pub via sealed box.
                 If source_name present, an extra per-record key wraps a C4 secretbox over source_name.
[POST] /concerns { title_ct, body_ct, source_name_ct?, hazard_class, severity, location_id, anon: bool }   [B2]
[Server] RLS denies INSERT unless is_active_member(); INSERT row; audit log {action='concern.create'}.
[List view] Members SELECT (RLS pre-filtered); client decrypts blobs per row.
[Source reveal path] Member clicks "reveal source"; client prompts per-record passphrase (C4);
                     audit log {action='source.reveal'} appended BEFORE the plaintext is shown
                     (so even an aborted reveal is recorded).
```

### 2.3 Export to employer co-chair (T11 + T12)

```
[Auth + role] Acting principal MUST be worker_co_chair (RLS gates the read of finalized minutes).
[Field selection (whitelist)] Export builder reads from a hard-coded allowlist of column refs
  for the document type: {minutes.final: title, agenda_items, decisions, attendees(committee_role only)}
  {recommendations.export: title, body, sent_date, employer_response?}
  source_name is NOT in any allowlist; the allowlist is closed (no "..." spread).
[Client decrypt] Fetch ciphertext blobs; decrypt in-browser with ck_priv.
[Interstitial / sign-off]
  - List every included field by label (not value).
  - Banner: "This will leave the worker side. Continue?"
  - Co-chair confirms by re-asserting passkey (re-auth, not just click-through).
  - If practical, 4-eyes: a second worker member's passkey assertion required.
[Render] In-browser PDF (jsPDF or pdf-lib) -> Blob -> URL.createObjectURL -> <a download>.
  NO POST to any server-side renderer. NO Edge Function involvement in PDF bytes.
[Audit] BEFORE the blob is offered to the user, POST audit log
  {action='export.minutes' OR 'export.recommendation', target_id, recipient_role='employer_co_chair',
   field_set_hash, ts, approver_id?}
  on failure to log -> abort the export.
[Off-app delivery] User downloads PDF, delivers via email/paper. App is not the transport.
```

### 2.4 Reprisal log (T13)

```
[Write] Member -> form -> client encrypts body with ck_pub (sealed box) AND a per-record symmetric key
        wrapped to ck_pub (C4 second layer). Per-record passphrase is required to view, not to write.
[Read] Author / co-chair / certified member only. Client retrieves ciphertext, unwraps with ck_priv,
       then prompts per-record passphrase to unwrap the C4 inner layer.
[Sensitive-read notification] On unwrap (T11 surface), client POSTs audit log {action='reprisal.read'}
       BEFORE rendering plaintext. The "recent sensitive reads" view subscribes to this stream and
       surfaces to every active member.
[Delete] Two-member quorum via `pending_destructive_ops` table. Member A proposes; member B (distinct)
       approves; only then RLS permits DELETE. Audit log captures both approvers.
[Retention] Active matter + 7 years from last-activity; real delete on schedule; crypto-shred backstop
       when committee key rotates and the old wraps are destroyed.
```

### 2.5 Auth + session model (T05, ADR-0002)

```
[First-device enrollment]
  Co-chair generates TOTP invite slip (paper or email).
  User loads /enroll; enters TOTP; server validates; user binds first passkey via WebAuthn (platform or roaming).
  Server destroys TOTP secret atomically with passkey binding (single transaction).
[Subsequent device]
  Existing passkey-authenticated session approves the new authenticator (no TOTP re-issue).
[Session]
  Access token TTL <= 15 min. No long-lived refresh token. Re-auth on resume via passkey assertion.
[Revocation]
  Settings -> "Sessions": list of (device fingerprint, last-seen, current?), revoke per row or all.
  Revoke = server invalidates JWT family + posts audit log.
[Recovery]
  No passkey + no other enrolled device -> co-chair issues fresh TOTP invite (4-eyes if multiple co-chairs);
  identity-key rotation may follow (ADR-0002 + ADR-0003).
```

## 3. STRIDE analysis per component

Format per finding: **F-id | STRIDE | description | affected component | current mitigation (ADR/task ref) | residual risk | testable mitigation**.

### 3.1 Priority 1 — E2EE key handling (T07, ADR-0003)

#### F-01 | S | Forged committee-key wrap insertion by a co-opted member
A member with valid auth re-wraps `ck_priv` for a non-member identity public key (e.g., the attacker's own off-roster account, or a key the employer holds). Server only sees a wrap for member_id X; if X is not currently active, the wrap is dead, but if the attacker can also create a stub `users` row or flip an inactive member to active, the wrap becomes live.
- **Affected:** `committee_key.wrap` insert path (T07), `committee_membership` admin (T06).
- **Mitigation:** Co-chair-only INSERT on `committee_membership`; active flag managed by co-chair (RLS) (ADR-0004, T06). Invite flow requires an existing TOTP-issued account (T05).
- **Residual:** Medium — co-chair takeover (A3) can chain to this.
- **Testable mitigation (T07):** *"A wrap row whose `member_id` does not have `committee_membership.active = true` at the time of INSERT must fail RLS. Test: as member M, attempt to insert a wrap for user U where U is inactive; assert RLS denies. Repeat with U not in `committee_membership` at all; assert RLS denies. Repeat with U made active by co-chair after wrap insert; assert wrap was rejected on the insert and is not retroactively valid."*

#### F-02 | S | Spoofed identity public key on first enrollment
A user enrolls and posts an identity public key that is not the one paired with the locally-held private key (e.g., MITM on the enrollment POST, or a buggy client). Subsequent wraps to this `ident_pub` cannot be opened by the legitimate user; effectively a denial-of-access. If the attacker controls the corresponding private key, the attacker decrypts.
- **Affected:** `users.identity_pubkey` insert (T05/T07).
- **Mitigation:** TLS 1.3 + Supabase Auth JWT bound to the request. No second factor verifying public-key/private-key pairing.
- **Residual:** Medium — TLS protects transit but the server has no way to verify pubkey-privkey pairing.
- **Testable mitigation (T07):** *"At identity-key enrollment, client must perform a self-test challenge: server returns a nonce sealed to the just-posted ident_pub; client must unseal and return; if unseal fails, enrollment is rolled back and the row is deleted. Test: register an ident_pub whose privkey is on a different keypair; assert enrollment is rejected within one request-cycle and no `users.identity_pubkey` row persists."*

#### F-03 | T | IndexedDB identity-private-key blob tampered by malicious extension or browser plugin
A browser extension with broad permissions modifies `ident_priv_wrapped_local` in IndexedDB (replaces with attacker's wrap). On next session, user re-derives K_p via passkey; the local wrap fails to open OR opens to the attacker's chosen privkey.
- **Affected:** IndexedDB at-rest (B4).
- **Mitigation:** No mitigation in stack today beyond the OS+browser sandbox. Recovery-passphrase blob on server is unchanged.
- **Residual:** Medium-low — requires malicious extension installed on the user's PWA origin context.
- **Testable mitigation (T07):** *"On every session start, after unwrapping `ident_priv_wrapped_local`, the client must perform a self-test: decrypt a known sentinel (a fixed sealed box to ident_pub, stored on the server, immutable post-enrollment). If self-test fails, the client must (a) refuse to proceed, (b) prompt recovery-passphrase flow, (c) post a `client.identity_selftest_fail` audit-log entry. Test: corrupt the IndexedDB blob; assert client refuses to enter authenticated state and surfaces the recovery flow."*

#### F-04 | T | Rotation race: two members rotate simultaneously
Two members each take the (advisory) lock-free path of `committee_key/rotate`; the second overwrites the first's wraps, possibly with a key the first member has already used to encrypt new C3 writes against the stale public key.
- **Affected:** Rotation endpoint (T07).
- **Mitigation:** ADR-0003 invariant 6 — Edge Function takes a per-committee advisory lock.
- **Residual:** Low — invariant 6 covers it IF the test confirms it.
- **Testable mitigation (T07):** *"Concurrent rotation calls must serialize: spawn two simultaneous rotation requests for the same committee from different members; assert exactly one succeeds and the other returns HTTP 409 (lock held) within the lock TTL. Assert no `committee_key` row exists with a wrap set that mixes pre- and post-rotation wraps. Assert no transient state where `committee_key.public` is updated but wraps for remaining members are missing."*

#### F-05 | T | Removed member's wrap not fully purged
On removal of member X, X's `committee_key.wrap` row is supposed to be deleted. If the rotation is partial (writes succeed but DELETE fails), X retains decryption capability for OLD ciphertext until backup expiry.
- **Affected:** Rotation flow (T07).
- **Mitigation:** Invariant 6 atomicity.
- **Residual:** Medium without a test.
- **Testable mitigation (T07):** *"After rotation triggered by removal of member X, X's row in `committee_key` (current AND history) is absent. Test: remove X; query Postgres directly (admin connection) for any row in `committee_key*` with `member_id = X`; assert zero rows. Also assert X's session is invalidated within the same transaction (no race between rotate-complete and session-revoke)."*

#### F-06 | T | Forward-secrecy gap: backups still contain X's wrap
Even after rotation, the nightly `pg_dump` for the prior 35 days contains X's old wrap and so could decrypt OLD ciphertext IF the backup-encryption key AND X's identity key are both available. Backups are encrypted with an escrow key; X does not hold the escrow key; but a subpoena to Backblaze + an escrow-key compromise + X's cooperation chains the breach.
- **Affected:** Backup chain (T17, B5).
- **Mitigation:** Backup is itself a libsodium secretbox; the bucket holds ciphertext-of-ciphertext.
- **Residual:** Medium — documented as accepted in ADR-0012 ("by design, T1/T5").
- **Testable mitigation (T17):** *"`pg_dump` files in the bucket must not be readable without the escrow key. Test: pull a fresh dump from the bucket via the bucket's own admin credentials; attempt to open with libsodium secretbox using a random key; assert decrypt fails. Test the dump-key never appears in the same provider's metadata, tags, or object ACLs."*

#### F-07 | R | Re-wrap or rotation lacks attribution
Existing audit-log spec covers role changes and concern actions; an explicit "wrap added/deleted/rotated" event is not enumerated in the failure-mode analysis or PI inventory. Without it, a later forensic question ("who re-wrapped for member N? when?") cannot be answered.
- **Affected:** Audit log (T18) ∩ rotation flow (T07).
- **Mitigation:** None today — gap in architecture.
- **Residual:** Medium — repudiation surface for A3.
- **Testable mitigation (T07 + T18):** *"Every key-material mutation (committee_key insert, wrap insert, wrap delete, rotation) writes an audit-log row with action ∈ {'key.committee.create','key.wrap.add','key.wrap.delete','key.committee.rotate'} containing the actor, target member_id, rotation_id (where applicable), and prev-hash. Test: trigger each of the four operations and assert the audit-log row exists, chains correctly, and is visible to all active members."*  **FINDING:** architecture gap; surface to architect as ADR follow-up.

#### F-08 | I | Recovery-passphrase backup blob brute-force
`users.identity_privkey_recovery_blob` is on the server (C2 ciphertext). If an attacker exfiltrates it (A5) and the user chose a weak passphrase, offline brute-force is possible.
- **Affected:** Recovery-blob storage (T07).
- **Mitigation:** Argon2id KDF with high parameters; passphrase chosen client-side and required to be typed-back to confirm.
- **Residual:** Medium if Argon2 parameters are weak.
- **Testable mitigation (T07):** *"Argon2id parameters MUST be: ops=4, mem=2^19 (512MB) OR a documented per-user override floor. Test: capture an emitted recovery-blob; verify the embedded KDF parameters meet the floor. Bench a brute-force attempt of 10 candidate passphrases; assert wall-clock per attempt > 250ms on CI hardware."*

#### F-54 | I | Recovery-passphrase low-vision escape hatch — on-screen "show again" creates a new visibility window beyond the printed sheet
**Cross-ref:** a11y-review A-5; design-system §4.D (Surface D, recovery-passphrase enrollment, D.4 generate / D.5 print / D.6 type-back verify); F-08 (recovery-blob KDF + type-back is the mitigation), F-41 (printed sheet coercion already accepted as O-1 outer-edge); ADR-0003 invariant 2 (no admin recovery).

**Description.** Surface D.6 hides the just-shown passphrase and requires the user to consult the printed sheet (D.5) when typing it back. Design-system §4.D today says the *only* path back to a re-display is "3 wrong attempts → return to D.4". A11y A-5 correctly identifies that a worker with low vision, dyslexia, or cognitive impairment may not be able to read the printed mono passphrase at all, and the "fail 3 times to re-display" path is punitive — it forces the user to deliberately enter incorrect input to gain access to an accommodation they need.

A11y A-5's proposed fix is a "I can't read the printed sheet — show the passphrase again" secondary link that returns to D.4 (re-display). The accessibility-specialist already routed this to the threat-modeler with their own preliminary assessment: "the coercion vector is the printed sheet itself (F-41), and re-displaying on demand is no worse than the existing 3-wrong-attempts path." This finding is the threat-modeler's confirmation pass plus added test obligations.

**Threat space (STRIDE-walked).**

  *Spoofing — N/A.* Nothing about identity assertion changes.

  *Tampering — N/A.* The passphrase is generated client-side and not transmitted; re-display does not change tamper surface.

  *Repudiation — Low.* If "show again" emits an audit-log row (M-54b below), the user cannot later claim the passphrase was never displayed; if it does not, there is no record. Audit-log row required.

  *Information disclosure — the load-bearing concern.* Three sub-threats:
    - **I-54.1 Shoulder-surfing during "show again".** A rep with low vision is statistically more likely to enroll in a setting where another person is present to help them (a co-chair, a colleague, sometimes a supervisor). The on-screen "show again" surface is shoulder-surfable; the printed sheet, once printed and folded, is not. The "show again" path therefore creates a **new** visibility window beyond F-41.
    - **I-54.2 Browser extension / DOM read during "show again".** A malicious extension with `activeTab` permission reads the passphrase from the DOM while it is rendered. The print stylesheet renders to a separate print pipeline (the browser's print stream); a malicious extension can intercept `window.print` calls and capture content, but that surface already exists in F-41-adjacent territory and is bounded by extension permissions. The on-screen "show again" surface is **simpler** to read from the DOM (no print-event interception needed) — so it widens the extension attack surface by a small but real amount.
    - **I-54.3 Coerced read-aloud.** A coerced rep is forced by an attacker to invoke "show again" and read the passphrase aloud. The attacker did not need to ask the user to retrieve the printed sheet (which the user may have stored away from work, per design-system §4 D.4 body copy). The on-screen surface defeats the "leave the printed sheet at home" mitigation by giving the attacker a way to extract the passphrase from a worker who is sitting at the device.

  *Denial of service — Low.* If "show again" is rate-limited (M-54c), a hostile script can lock the enrollment by exhausting attempts; mitigation is the per-enrollment-session counter, after which the user must restart enrollment.

  *Elevation of privilege — N/A.*

**Mitigation options considered (per accessibility-specialist's enumeration) and threat-modeler verdict on each.**

  **(A) Audio-narration mode (OS TTS reads the passphrase, headphones recommended).** *Verdict: WIDENS threat space.* The audio channel is leakier than the visual channel in a workplace setting — headphones may not be available, ambient microphones (the device's own, a nearby coerced rep's phone, an MDM screen-recording-with-audio) capture audio; "recommended" is not enforceable. Audio also defeats the F-41 "don't store at work" mitigation more decisively than visual re-display (a coerced rep would not need to even look — they could be made to listen while the attacker records). **Reject.**

  **(B) Larger-font display with high-contrast on demand.** *Verdict: ACCEPTABLE; smallest threat-space delta.* This is a magnifier applied to the same on-screen surface the user already had at D.5 (print layout is shown on-screen as a fallback when `window.print` is unavailable, per current D.5 spec — so on-screen rendering of the passphrase already exists in the flow). Increasing font size or contrast does not add a new surface; it adjusts an existing one. The shoulder-surf threat (I-54.1) is mathematically the same as a user with a magnifier on the printed sheet — a real but pre-existing threat. **Acceptable, but on its own does not solve A-5 because the user must still be on D.5 to use the magnifier; A-5's specific case is on D.6 where the passphrase has been hidden.**

  **(C) "Show for N seconds with hold-to-reveal" (password-reveal UX).** *Verdict: ACCEPTABLE; narrows the threat space versus a permanent re-display.* The hold-to-reveal pattern bounds the visibility window to the user's active intent (the user holds a touch / pointer / Space-key on the reveal button; releasing hides). Shoulder-surfing (I-54.1) is bounded to the hold duration; DOM-read by extension (I-54.2) is still possible but only during the hold; coerced read-aloud (I-54.3) is bounded by the user's willingness to hold (a coerced rep can be made to hold, so this does not fully defeat the threat — but it does emit an audit-log row each time per M-54b, which limits silent extraction).

  **(D) Require a second device to read on while typing on the laptop.** *Verdict: NOT VIABLE.* Adds friction and presumes a second device the worker may not have. Also, the second device's screen is itself shoulder-surfable, and "use your phone" implicitly tells the user to store secrets on their phone — which is the device most likely to be MDM-enrolled in BYOD employer programs. **Reject for v1.**

  **(E) Skip type-back verification.** *Verdict: NOT VIABLE.* F-08 type-back is the mitigation that ensures the recovery-blob's passphrase actually matches what the user can type back; skipping it converts a recovery procedure that has been verified-once into a recovery procedure that has been generated-but-not-verified, raising the risk of locked-out users at the worst time. **Reject.**

**Chosen path: combination of (C) hold-to-reveal AND (B) on-demand magnification, gated by audit-log emission AND a per-enrollment-session cap.** Justification: (C) is the strongest single control against I-54.1 and I-54.2 because it bounds the visibility window; (B) is the smallest-delta accommodation for users whose A-5 issue is "I cannot read the standard font even when displayed". Together they provide an accessibility path WITHOUT a permanent on-screen surface. The audit-log emission and the session cap protect against silent or repeated extraction.

**Affected:** Surface D.6 (and the secondary link returning to a constrained D.4-variant); recovery-passphrase enrollment flow (T07). Code paths: `src/lib/onboarding/recovery/*` (when implemented), the print stylesheet (already at D.5), the WebAuthn-bound enrollment session state machine.

**Likelihood:** Low-to-medium for I-54.1 (workplace shoulder-surf is plausible, mitigated by hold-to-reveal). Low for I-54.2 (requires installed malicious extension). Medium for I-54.3 (coercion is the in-scope adversary model A3).

**Impact:** High — recovery passphrase unwraps `users.identity_privkey_recovery_blob`, which yields the identity privkey, which unwraps the committee key, which decrypts all C3/C4 the user could see. Same impact envelope as F-08 + F-41.

**Priority:** Medium (the type-back verification's existence narrows the practical exploit window — enrollment is a single moment, and the F-41 printed-sheet vector dominates the long-tail risk).

**Testable mitigation:** Four invariants.

  *Invariant M-54a — Hold-to-reveal gating.* "The 'show passphrase again' control on D.6 (the secondary link from A-5) MUST be a hold-to-reveal: the passphrase is revealed only while a pointer-down OR Space-key-down OR Enter-key-down is sustained on the reveal control for ≥1500ms continuously, and is hidden immediately on release. The reveal control's label MUST name the consequence in plain language (catalog string, e.g., `onboarding.recovery.show_again.label` = 'Hold to show the passphrase on screen'); the helper text MUST name the threat (catalog string, e.g., `onboarding.recovery.show_again.helper` = 'Anyone who can see your screen right now will see the passphrase.')."
  - **Assertion shape:** *"Render Surface D.6; click the 'show again' control with a normal click (release within 100ms); assert the passphrase is NEVER rendered in the DOM (no element with `data-testid='recovery-passphrase-onscreen'` becomes visible). Hold for 1500ms; assert the passphrase IS rendered after 1500ms. Release; assert it is removed within 50ms. Hold for 5000ms; assert continued visibility until release. Test against keyboard: Space-keydown for 1500ms reveals; Space-keyup hides."*

  *Invariant M-54b — Audit-log emission on every reveal.* "Every successful invocation of 'show passphrase again' MUST emit a `identity_privkey.recovery_blob.viewed` audit-log row BEFORE the passphrase becomes visible in the DOM, with `{actor_id, enrollment_session_id, reveal_count_in_session, ts}`. The audit-log row is server-instrumented (the reveal control invokes a server-side log endpoint and only renders the passphrase after the log endpoint returns 200). Pre-enrollment users have a special enrollment-scoped logging path (the audit log row is committed under the partially-enrolled user's id, which is bound to the TOTP-invite-consumed session)."
  - **Assertion shape:** *"Trigger 'show again' once successfully; assert exactly one audit-log row with action='identity_privkey.recovery_blob.viewed' for the current user with the correct `enrollment_session_id`. Mock the audit-log endpoint to return 500; trigger 'show again'; assert the passphrase is NOT rendered (no DOM element appears) AND a `toast.danger` is shown to the user. Trigger 'show again' three times successfully; assert three audit rows with `reveal_count_in_session` = 1, 2, 3."*

  *Invariant M-54c — Per-enrollment-session cap of 3 reveals.* "The 'show passphrase again' control MUST NOT be invocable more than 3 times within a single enrollment session. On the 4th attempt, the control is `aria-disabled=true` with a visible helper text directing the user to restart enrollment from D.1 (which generates a fresh passphrase, fresh recovery blob, fresh print). Restart-enrollment emits its own audit-log row distinct from `viewed`. The 3-reveal cap resets on successful D.7 completion (i.e., it is a per-incomplete-enrollment cap, not a per-account cap)."
  - **Assertion shape:** *"Trigger 'show again' three times; assert all three succeed. Attempt a fourth; assert the control is `aria-disabled=true` AND no audit-log row is emitted AND a helper text directs to restart enrollment. Restart enrollment; assert a fresh enrollment session id; assert 'show again' is invocable again (up to 3 times in the new session)."*

  *Invariant M-54d — On-screen reveal is the ONLY accommodation path; no audio narration; no clipboard copy from the reveal control.* "The 'show again' surface MUST NOT offer a clipboard-copy button on the revealed passphrase (the user already had clipboard copy at D.4 once, per design-system D.4 secondary action; the 'show again' surface does NOT re-expose it because clipboard remains a coercion / exfil channel). The 'show again' surface MUST NOT invoke `SpeechSynthesisUtterance` or any TTS API. The browser's own OS-level zoom + the on-demand high-contrast (per Blocker B-1's tertiary token usage and a11y accommodations) are the supported accommodations for low vision; the implementer MUST NOT add an in-app TTS path."
  - **Assertion shape:** *"Render Surface D.6 reveal state; assert no element with `data-testid='copy-passphrase'` is present in the DOM in this state (only at D.4, not at the D.6 reveal). Static lint: grep the recovery-flow modules for `SpeechSynthesisUtterance`, `window.speechSynthesis`, `tts`; assert zero matches outside test fixtures."*

**Interaction with F-41 (printed-sheet coercion).** F-41 is already accepted as O-1 (duress mode is v2). The "show again" surface does NOT widen the F-41 acceptance because: (1) the printed sheet remains the canonical secret with the same coercion profile; (2) on-screen reveal is bounded to the enrollment session window (post-enrollment, the surface is gone — the user cannot invoke "show again" from settings, because there is no settings surface for it; if they need the passphrase post-enrollment, they consult the printed sheet); (3) the audit-log row (M-54b) makes silent post-enrollment extraction impossible by definition. **The widening is limited to the enrollment session itself**, which is a small temporal window in the user's lifecycle, and the hold-to-reveal + cap + audit-log bound the damage.

**FINDING:** Surface D.6 must gain a "show passphrase again" path per A-5 with the four invariants M-54a/b/c/d. Add a new enum value to the audit-log action set: `identity_privkey.recovery_blob.viewed`. Cross-link with O-1 (F-41) — the duress-mode-v2 work, when it lands, should also revisit the "show again" surface and consider whether to remove it entirely under duress mode.

#### F-53 | E | Protected-modal focus-trap timing race during opacity transition (export_interstitial + 4 other protected modals) — RA-1 compensating-control bypass
**Cross-ref:** a11y-review A-2; design-system §3.1 (focus rules), §3.2 (protected modal variants), §4.A (export interstitial states `re-auth-required` → `reviewing-fields` → `concern-flag-warning` → `confirming`); RA-1 in `.context/decisions.md` (single-signer co-chair re-auth on export accepted with concern-derived-flag + allowlist + audit + post-export rep notification as compensating controls); F-19 (closed allowlist), F-24 (audit-row precondition), F-29/HG-1 (4-eyes posture). Applies to all five protected modal variants but the highest-impact instance is **`export_interstitial`** (B3 egress) and **`reauth_prompt`** (the RA-1 single-signer gate).

**Description.** The design system in §3.1 says "Modal opens: focus moves to the first focusable element inside the modal, focus trap engages, body scroll locks" but does not pin **when** in the modal lifecycle the focus trap and the `aria-labelledby` announcement engage. The implementer could legitimately read this as "engage on `transitionend` of the opacity transition" because the visible-completion model is what most modal libraries default to. The `motion.per_surface.modal_enter` token uses `micro` (100ms in reduced-motion) and `normal` (~200ms otherwise) opacity transitions. That window — between `modal.show()` and `transitionend` — is the race surface.

An attacker on the worker's device, operating in the same SPA context (one of: compromised browser extension with broad host permissions, malicious clipboard handler scripted to fire keystrokes, coerced rep running a Selenium-style helper, or a remote DevTools attach via Chrome Remote Debug) can:
  1. **Race Tab+Enter on the underlying Confirm button.** If the user opens the export interstitial from a surface that has a stale focus on a button labeled "Confirm" or "Submit" or "OK", and if the focus trap engages only on `transitionend`, then a synthesized `keydown:Enter` within the 100–200ms window can be received by the surface BENEATH the (still-mounting) modal. The scrim at `z_index.scrim` catches **pointer** events, but `keydown` is delivered to the focused element, which has not yet been moved by the trap.
  2. **Synthesize the modal's own primary action before the announcement promise resolves.** Suppose the focus trap moved focus on `modal.show()` but the announce-on-open promise (the screen-reader-friendly `aria-labelledby` announcement) is fired only after `transitionend`. A scripted `keydown:Enter` on the now-focused primary button confirms the action before the SR user has heard the announcement — which is exactly the coercion-resistance posture the modal is supposed to enforce (the SR user must hear *what they are about to do* before the action goes through).
  3. **Synthesize Escape during the transition.** §3.2 says these five modals do not dismiss on Escape — but the **`keydown:Escape` handler binding** is part of the modal's own lifecycle. If the handler is installed only after `transitionend`, an Escape during the transition window propagates to the document, which may have a global Escape handler (e.g., a parent surface's "go back" affordance) or no handler — either way, the modal silently does not fire its "explicit Cancel required" behavior.
  4. **Click-on-backdrop during transition.** The scrim catches pointer events at full opacity, but during a 100–200ms fade-in the scrim's `pointer-events` may be `none` or its opacity may be below the implementer's threshold. A click during the transition can land on the underlying surface.

**Affected:** All five protected modals (`export_interstitial`, `reauth_prompt`, `passphrase_prompt`, `destructive_confirm`, `four_eyes_pending`) — design-system §3.2. Most consequential: export interstitial (T11/T12, B3 egress); reauth_prompt (RA-1 single-signer gate on export); passphrase_prompt (C4 read gate on Surface C reprisal log + F-34 friction layer). Code paths: any modal mount in `src/lib/ui/modal/*` (when implemented), every call site in Surfaces A, C, D, G, I.

**Mitigation status today.** Design system §3.1 and §3.2 are **ambiguous** on the trap-engagement timing. The a11y reviewer correctly flagged the ambiguity (A-2) and noted it is "advisory" from a strict-WCAG standpoint, but explicitly routed it to threat-modeler because the security posture (coercion-resistance) raises the stakes from advisory to load-bearing.

**Does this materially change RA-1?** **Yes.** RA-1 accepted single-signer co-chair re-auth for export because four compensating controls are present: (1) the recipient is visible plaintext in the interstitial, (2) the closed export allowlist (F-19) blocks C4 fields structurally, (3) every export is audit-logged before the blob is offered (F-24), (4) every active member receives a post-export sensitive-activity notification. Controls (1), (3), and (4) all assume the user CONSCIOUSLY confirmed the export — i.e., that the user heard or saw what they were confirming before the confirm fires. A 100–200ms race during which the modal is visually present but functionally unguarded weakens control (1) (the user has not yet heard the recipient announced) and weakens the "F-24 audit row reflects user intent" claim (the audit row records a confirm action that the user did not consciously perform). The export still happens through the allowlist (F-19 holds — that's structural), so plaintext source_name still does not leak via this path. **But the RA-1 acceptance rests on more than F-19; the consent-shape of the confirm matters.** If the protected modal can be confirmed in <200ms by a scripted attacker, single-signer re-auth becomes meaningfully weaker than the ADR's risk assessment assumed, and the case for HG-1 leaning toward 4-eyes strengthens.

**Likelihood:** Medium for an attacker with extension privileges or a coerced-rep + helper-script scenario (A2 + A3 in the adversary model). Low for the conventional A4 external attacker. Reachability is bounded by the requirement that the attacker have script execution in the SPA origin (which is exactly what a malicious extension or an MDM-injected script accomplishes — both are in-scope adversaries).

**Impact:** High when the modal is `export_interstitial` (B3 egress; the export is the only routine egress for C3 data, and a scripted confirm bypasses the RA-1 compensating-control posture); medium for `reauth_prompt` (escalates a session to a privileged action without the user's conscious assent); medium for `passphrase_prompt` on Surface C (C4 read without the user's conscious assent — but the read itself is still server-instrumented per F-33, so the audit row still fires; the harm is "the SR user did not hear what they were about to read").

**Priority:** High (med-high likelihood × high impact for the export case).

**Testable mitigation (T11 + T12 + design-system contract):** Three invariants, each independently testable.

  *Invariant M-53a — Trap-engages-on-mount.* "For every modal variant in the protected set {`export_interstitial`, `reauth_prompt`, `passphrase_prompt`, `destructive_confirm`, `four_eyes_pending`}, the focus trap, the Escape-handler binding, the click-outside-no-op binding, and the `aria-modal=true` / `aria-labelledby` announcement MUST be installed synchronously with `modal.show()`, before the next animation frame, and BEFORE any opacity transition begins. The opacity transition is decorative-only."
  - **Assertion shape (test-writer must write):** *"Mount a protected modal with `motion.duration.normal` opacity transition set to 1000ms (artificially extended) and `motion.duration.micro` set to 1000ms in reduced-motion mode. At t=10ms after `modal.show()` (well within the transition), synthesize a `keydown:Enter` on `document.activeElement`. Assert: (a) `document.activeElement` is inside the modal (not on the underlying surface); (b) the modal's own primary action did NOT fire (the announce-on-open promise has not resolved yet — see M-53b); (c) no audit-log row was written; (d) repeat with `keydown:Escape`; assert the modal's own Cancel behavior is invoked (for the four variants that have Cancel) OR is a no-op (for `four_eyes_pending` which has no Cancel — assert Escape is swallowed). Repeat with synthesized click at the coordinates of the underlying surface's primary button; assert no event reaches the underlying surface (scrim has `pointer-events: auto` from t=0)."*

  *Invariant M-53b — Announce-on-open precedes input acceptance.* "Protected modals MUST NOT accept any keyboard or pointer input that resolves to a primary or secondary action until the announce-on-open contract has resolved. The announce-on-open contract is: `aria-modal=true` is set, `aria-labelledby` is wired to the headline element, and the headline element is present in the accessibility tree. The implementer represents this as a `ready` promise; primary/secondary action handlers MUST await this promise before dispatching."
  - **Assertion shape (test-writer must write):** *"Stub the modal's `ready` promise to resolve only after a controlled delay (200ms). At t=10ms after `modal.show()`, synthesize `keydown:Enter` on the modal's primary button. Assert: the primary action's handler does NOT fire — either because the button is `aria-disabled=true` until ready, or because the handler awaits `ready` and the assertion captures the handler entering a pending state. At t=210ms (after ready resolves), synthesize again; assert the handler now fires. For `export_interstitial` specifically: assert that any confirm before `ready` resolves results in NO `export.minutes` / `export.recommendation` audit-log row AND no Blob URL creation."*

  *Invariant M-53c — Underlying surface inert from t=0.* "When a protected modal mounts, the underlying surface MUST be marked `aria-hidden=true` AND `inert` (or polyfilled equivalent) AND its focusable descendants MUST have `tabindex=-1` applied synchronously with `modal.show()`. The scrim MUST have `pointer-events: auto` and a `keydown` capture-phase handler that swallows Tab/Enter/Space/Escape targeted at the underlying surface during the transition."
  - **Assertion shape (test-writer must write):** *"With a protected modal mounted but its opacity transition artificially extended to 1000ms, attempt to focus a button on the underlying surface via `element.focus()` (programmatic, simulating an attacker-controlled extension); assert `document.activeElement` is NOT that button. Attempt to `dispatchEvent(new KeyboardEvent('keydown', {key:'Enter'}))` against an underlying surface button; assert the button's `click` handler does NOT fire. Pointer-event test: dispatch a `click` at the coordinates of the underlying surface's primary button; assert `event.defaultPrevented === true` or the click target is the scrim, not the button beneath."*

### 3.4 Priority 4 — Reprisal log (T13)

#### F-30 | S | Reprisal-log read by removed-but-cached-session member

- **Mitigation:** ADR system design says reads visible to all members; failure mode T11 test covers it.
- **Residual:** Medium without server-side enforcement.
- **Testable mitigation (T13 + T18):** *"Reading a `reprisal_log` row is server-instrumented: an Edge Function (or RLS-via-view trigger) writes the `reprisal.read` audit-log row on every successful SELECT, not just on client-initiated logging. #### F-34 | I | Per-record passphrase weakness
The per-record passphrase for C4 is freely chosen by the recording member; weak passphrase + breach of `committee_key.wrap` yields plaintext.
- **Affected:** C4 per-record key wrapping (T13).
- **Mitigation:** Per-record key is symmetric and wrapped to the committee public key; the passphrase only unlocks a *client-side* prompt before render (not the cryptographic gate).
- **Residual:** Low if the architecture is "passphrase is UX gate, not crypto gate." Medium if implementer treats it as the crypto gate.
- **Testable mitigation (T13):** *"The per-record passphrase is a UX confirmation only; the cryptographic gate is `ck_priv` + the per-record key wrapped to `ck_pub`. The passphrase requirement is a friction layer that DOES log the read attempt (F-33) even when wrong. Test: enter wrong passphrase three times; assert plaintext NOT shown AND a `sensitive.access_attempt` row written."*  **FINDING:** clarify semantics in T13 spec.

#### F-35 | D | Reprisal-log feed flooded by spurious entries
A rep submits many fake entries to overwhelm the committee's attention.
- **Affected:** `reprisal_log` INSERT (T13).
- **Mitigation:** Rate limit + audit log + 4-eyes on DELETE (deletion gated, but submission isn't).
- **Residual:** Low — friction is the right level here; reprisal needs to be writeable freely.
- **Testable mitigation (T13):** *"Rate limit on `reprisal_log` INSERT: <= 10 per actor per hour. Test: 11th returns 429."*

#### F-36 | E | Co-chair-only delete bypassed via direct UPDATE to set tombstone
A co-chair sets `status='deleted'` directly when DELETE is gated; if any retention job or list view filters by `status != 'deleted'`, the effect is a silent delete with no 4-eyes.

### 3.9 §3.6 Retention sweep (T16 library + T16.1 SQL — naming honours ADR-0017 §10 brief; numbered §3.9 because §3.6–§3.8 are taken in this file's existing structure)

Source: ADR-0017 (T16 retention sweep library + MemoryRetentionStore), 2026-05-23. Closes the F-51/F-52 surfaces with concrete library-level test obligations and adds five new surfaces specific to the sweep mechanism (lease, row-cap, schedule-hash, TestStore split, no-PII-in-errors).

**Scope split:** F-55..F-69 below are scored as a family. Library-scope assertions land in T16 (CI tests against `MemoryRetentionStore`). SQL-scope assertions land in T16.1 (pgTAP, `SupabaseRetentionStore`, pg_cron, Edge Function trigger). Each row tags `[T16]` or `[T16.1 deferred]` for the test-writer.

#### Trust boundaries this introduces

- **B6 — Retention service-role boundary (added here; activates at T16.1; library shape preserved in T16).** A non-login `retention_service_role` in T16.1 holds `GRANT EXECUTE` on the `SECURITY DEFINER` sweep functions and is the only role authorised to issue cross-table DELETEs aligned to the schedule. It has NO `GRANT` on base tables — the only attack surface is the function set, and the function bodies hard-code the schedule lookup. The library half (T16) mirrors this at the type level: `RetentionStore` forbids caller-supplied WHERE; the predicate is constructed from the schedule discriminator only. Treat B6 as "type-level in T16, role-level in T16.1." Reference: ADR-0017 §1, §4, §6; sibling task spec items #1, #3, #9.
- **Diagram note (consolidated diagram, §1 of this file).** B6 sits inside the Supabase-cloud box, adjacent to "Postgres + RLS-as-authz" and "Edge Functions". Edge: `pg_cron` (T16.1) -> SECURITY DEFINER functions (owner `migration_role`, GRANT EXECUTE to `retention_service_role`) -> per-event-type and per-table DELETEs bounded by `RETENTION_SCHEDULE` / `OPERATIONAL_TABLE_SCHEDULE`. The audit-log `retention.deleted` summary row is the LAST write in the transaction (F-24 variant; see F-58). When the diagram is next refreshed, add a B6 box labelled "Retention service-role (non-login, SECURITY DEFINER bounded by closed schedule)" with arrows: pg_cron -> sweep fn -> DELETEs (per-event-type, per-table) -> retention.deleted (LAST) -> `retention_sweep_runs` checkpoint.

#### Data flows (T16 library shape; T16.1 SQL shape mirrors)

```
[Retention sweep pass — per ADR-0017 §6 algorithm]
  Caller (CI test in T16; pg_cron / Edge Fn in T16.1)
    -> runRetentionPass({store, config})
       1. store.findOpenRun(lease_window_ms)            // lease acquire (B6 entry)
          if existing -> return {status: 'lease_held'}  // F-59
       2. store.recordSweepRun({run_id, started_at_ms, status: 'running'})
       3. for each RetentionEventType in alphabetised order:    // F-55 closed-allowlist
            cutoff = resolveCutoff(RETENTION_SCHEDULE[event_type], nowMs)
            if event_type is 'retention.deleted': skip (F-62 carve-out)
            if kind is 'match_underlying':
              ask store.isTargetGoneFor(target_id, source_table)  // F-61 ceiling
              gate on (linked_record_deleted_for >30d)
            store.deleteForEventType(event_type, cutoff, batch_size)  // F-64 no WHERE
            accumulate per_event_counts
            check per_pass_row_cap                       // F-60 row-cap
       4. for each table in OPERATIONAL_TABLE_SCHEDULE:
            store.deleteForTable(table_name, predicate, batch_size)  // F-56 totp 24h
            accumulate per_table_counts
       5. if total > alarm_threshold (default 20):       // F-57 over-delete alarm
            result.alarm_fired = true (T16.1 emits A-RETENTION-001)
       6. summary = buildRetentionDeletedSummary({       // F-58 audit-WITH-side-effect
            run_id, started_at_ms, ended_at_ms,
            schedule_hash,                                // F-63 binds to schedule version
            per_event_counts, per_table_counts,
            prev_pass_hash, status })
       7. store.emitRetentionDeleted(summary)           // LAST write in tx
          if throws: ENTIRE TRANSACTION ROLLS BACK
                     (no deletes, no summary; F-58 invariant)
       8. store.closeSweepRun(run_id, {status, ended_at_ms, ...})  // F-68 RA-2 anchor
  Errors throughout: {run_id, status, error_code} only  // F-66 no PII

[Dry-run mode (CI test default per F-57)]
  step 3/4 use countCandidates* instead of delete*;
  summary is built but NOT emitted; result returns would-delete counts.
```

Classification per surface: `retention_sweep_runs` carries NO PI (counts + run metadata only; confirmed per ADR-0017 §8 of T16.1 deliverables). `retention.deleted` audit rows carry NO `target_id` (ADR-0015 carve-out) and store actor as the synthetic `retention_service` HMAC pseudonym. The sweep itself reads `audit_log.retention_class` (C1) and writes `audit_log` (C1); it never reads or writes C3/C4 plaintext. Residency: ca-central-1 (no new processor, no new cross-border flow — confirmed in ADR-0017 §Compliance check). Retention of the surfaces themselves: `retention_sweep_runs` = 7y (mirrors `retention.deleted`); `audit_log_retention_schedule` = permanent_schema_table.

#### F-57 math note (architect ask #1: drives F-51 Medium → Low?)

F-51's original residual was "Medium — high blast radius" because the sole control was "Crypto-shred for E2EE'd records; deletion logged; alert on volume" with no quantification of the volume threshold or default posture. F-57 generalises F-51 to ALL retention-eligible tables (not only `reprisal_log`) and stacks three controls:

1. **Dry-run as CI default.** Every CI run of the sweep is dry-run, so a regression that would mass-delete is caught BEFORE the live path executes. Likelihood of a bug reaching production with a delete-side-effect: high → low, because the implementer cannot land a green CI on a bug that would delete more than the fixture expects.
2. **Per-pass row-cap (default 20000).** Even a runaway sweep cannot delete more than `max_total_rows_per_pass` per execution. The blast radius is capped: at one pass per day (T16.1 pg_cron schedule), the maximum unrecoverable loss is bounded per 24h. Impact of a successful exploit / bug: high → medium, because hot recovery from a single pass is feasible from the backup (ADR-0012's pg_dump, 35-day retention) — the backup carries the rows the sweep deleted.
3. **Over-delete alarm at threshold 20 (configurable).** A pass whose deletes exceed 20 rows trips `alarm_fired = true`, emits the alarm via T16.1's A-RETENTION-001, and an operator reviews before the next pass at 03:30 ET. Operator latency: <24h in normal operation. Time-to-detect a runaway: ≤24h, often minutes once observability-setup wires the alert sink.

**Combined likelihood:** Low. A bug must (a) escape the drift test (F-55), (b) escape the dry-run CI test (F-57), (c) escape the alarm review window, AND (d) operate within the row-cap. The compound probability is at the "low" end across realistic deployment.

**Combined impact:** Medium. The row-cap bounds the per-pass blast radius to 20000 rows; the 35-day pg_dump window bounds recovery to the last 24h of writes-since-dump (acceptable per ADR-0015 §Risks "transient inconsistency is acceptable" posture). Not Low because a runaway hitting C4 audit metadata is still operationally painful; not High because the cap + backup combination is structural.

**Residual: Low.** F-57 drives F-51 down from Medium to Low when (1)+(2)+(3) all hold AND the T16.1 alarm wiring lands. Until T16.1 lands, residual is "Low (CI) / Medium (production-untested)" — the test obligation (T16-ALARM-1, T16-DRY-RUN-1) plus the architect-mandated production prerequisite (T16.1 A-RETENTION-001) carry the assertion.

#### F-58 invariant note (architect ask #2: F-24 inversion, not weakening)

F-24 in this file (export pipeline) reads "audit-BEFORE-side-effect": the export audit row commits before the Blob URL is created and the file leaves the worker's browser. The inversion intuition is "if the audit fails, the side-effect is harmless because nothing was disclosed." For retention sweep, F-24 takes a structurally equivalent form with the ordering INVERTED in time but the invariant PRESERVED:

- **The invariant (both F-24 and F-58):** *no side-effect commits without its audit row, and no audit row commits without its corresponding side-effect.*
- **F-24 (export):** audit emitted first, side-effect (Blob URL) follows. Failure mode = audit emit fails → no Blob URL.
- **F-58 (retention sweep):** side-effects (deletes) accumulate within an open transaction, summary row written LAST inside the same transaction. Failure mode = summary emit fails → ENTIRE transaction rolls back → no deletes commit AND no summary row. Per ADR-0017 §Option D: "the summary row is the LAST row written in the transaction (after the deletes) so its `meta.deleted_per_table` / `meta.deleted_per_event_type` counts are authoritative for what landed."

The audit-WITH-side-effect form is **stronger** than audit-BEFORE-side-effect for this surface because the summary row's counts are authoritative — they cannot understate what landed (the deletes already executed within the same tx) and they cannot overstate (a failed summary emit takes the deletes down with it). For the export surface F-24's "audit-BEFORE" is the right pattern because the export side-effect (file leaves the browser) is irreversible once it occurs; for the sweep, the deletes are reversible only within the open transaction, so the summary-LAST pattern guarantees atomicity.

**Verdict:** F-58 is a coherent F-24 variant, not a weakening. Test obligations T16-ATOMIC-1 (force summary throw → assert rollback) and T16-ATOMIC-2 (force mid-batch delete throw after summary built → assert rollback) pin the invariant directly. Per-batch savepoints are explicitly rejected (ADR-0017 §Option D) because they admit a partial-state failure mode that would weaken the invariant.

#### Architect ask #5 verdict — HG-10 NOT firing is defensible

PIPEDA Principle 4.5 (Limiting Retention) requires that PI be retained only as long as necessary to fulfil the identified purposes; it does **not** mandate pre-deletion notice to the individual. PIPEDA Principle 4.9 (Individual Access) requires that on request, an individual be given access to their PI — but it operates on the existing record, not on a future-deletion-pending record. ADR-0017's underlying-record-ceiling rule guarantees that audit rows expire AFTER the underlying record they reference, so a user querying their data while their membership is active sees the complete record; deletion at the floor is structural and follows the retention promise the user agreed to at consent time (the privacy policy / `.context/constraints.md` posture).

A pre-deletion notification surface would introduce: (a) a "deletion is pending; please confirm" interaction surface that ADR-0017 §Option F deliberately omits; (b) a new race between notification delivery, user response window, and the sweep's lease window; (c) a new attack surface for the employer adversary (A1) to subpoena a "pending deletion" snapshot. The threat-model concurs with ADR-0017 that v1 ships without pre-deletion notification and that HG-10 (if it existed for this purpose) does NOT fire.

If a regulator surfaces a pre-deletion-notice requirement post-launch, the carry-forward path is an ADR amendment re-firing HG-10 explicitly with new findings under §3.9. **Verdict: HG-10 NOT firing is defensible.** No BLOCK from threat-modeler.

## 6. Verification of the seven E2EE invariants (ADR-0003)

Each invariant is restated, mapped to a test, and (where applicable) strengthened.

### Invariant 1 — Server never sees a private key in the clear
**Restated:** Identity privates exist client-side or as ciphertext-on-server. Committee data privates exist only as wrapped blobs.
**Testable as written?** Mostly. Strengthen to include negative test against telemetry channel.
**Strengthened mitigation:** *"For every Edge Function and every Supabase function-log entry produced during the test suite, assert no byte sequence matches the format of an unsealed libsodium private key (32-byte high-entropy block in a context labeled `priv`/`secret`/`key`). Active assertion: T07 fixture inserts a known-canary 32-byte 'private-key-shape' value via DevTools into client state, triggers a flow; canary must never appear in any captured server-side log or Sentry payload."*  Tests: T07-test-1.

### Invariant 2 — No "admin recovery"
**Restated:** No server-side mechanism to recover a forgotten passphrase + lost device.
**Testable as written?** Yes, but the test is mostly an "absence" test (no endpoint exists).
**Strengthened mitigation:** *"Route inventory test (CI): no route under `/api/*` exists that accepts (user_id, request_admin_recovery) and returns plaintext identity material. No Edge Function accepts a passphrase-bypass token. Test: attempt to call hypothetical `/api/admin/recover-identity` and `/api/admin/decrypt-as`; assert 404. Manual: documentation explicitly states 'no admin recovery'."*  Tests: T07-test-2.

### Invariant 3 — No plaintext caching server-side, including in Edge Functions
**Testable as written?** Yes — combine with F-09.
**Strengthened mitigation:** as F-09; plus *"Semgrep rule bans `console.*(req)`, `console.*(req.body)`, `console.*(payload)` in Edge Function modules; runtime fixture posts a known-canary plaintext to an Edge Function; capture Supabase function logs; assert canary absent."*  Tests: T07-test-3.

### Invariant 4 — All encryption is libsodium primitives
**Testable as written?** Yes.
**Strengthened mitigation:** *"Semgrep rule: bans `crypto.subtle`, `node:crypto`, `bcrypt`, `aes-*` imports outside `src/lib/crypto/`; allowlists `libsodium-wrappers` only. CI test: full repo `grep -r` for forbidden imports; assert clean. Test: drop a forbidden import in a side branch; assert CI fails."*  Tests: T07-test-4.

### Invariant 5 — Key material in URLs is forbidden
**Testable as written?** Yes.
**Strengthened mitigation:** *"Semgrep rule: bans URL query params named `key`, `secret`, `passphrase`, `priv`, `nonce`; bans `Location: ...?...key=` patterns; bans setting cookies with names matching key patterns. Runtime test: spider every route; assert no URL contains base64-shaped 32+ byte tokens in query strings."*  Tests: T07-test-5.

### Invariant 6 — Rotation is atomic
**Testable as written?** Yes (mapped F-04, F-05).
**Strengthened mitigation:** as F-04 and F-05 combined. **PLUS: handle backup-window forward-secrecy gap (F-06) as an *accepted* residual, documented in §7.** Tests: T07-test-6a (concurrent rotation), T07-test-6b (X's wrap deleted), T07-test-6c (session revoke same transaction).

### Invariant 7 — C4 records require a second decrypt step logged in the audit log
**Testable as written?** Yes, but original wording is loose ("require a second decrypt step logged"). Tighten:
**Strengthened mitigation:** *"Reading a C4 row writes a `sensitive.read` audit-log entry BEFORE the plaintext is returned to the client (server-side enforcement via SECURITY DEFINER view or Edge Function indirection). Test: SELECT a C4 row directly with a valid member JWT; assert the SELECT either (a) returns the ciphertext and the read is server-instrumented to log on retrieval, OR (b) is funneled through a view/function that requires a logging step. Assert the audit-log row appears within the same transaction as the read."*  **FINDING (cross-link F-33):** original wording invites client-cooperative logging. Tighten to require server-side enforcement.  Tests: T07-test-7 + T13-test-3.

### Additional invariant to add — Invariant 8 (proposed): Key-material mutation audit trail
**Proposed wording:** *"Every mutation of key material (committee_key insert/update; wrap insert/delete; rotation) emits an audit-log row with a defined enum action and hash-chains to the prior audit row."*  Justification: F-07 surfaced this gap. **HUMAN-GATE DECISION:** add Invariant 8 to ADR-0003 (lightweight ADR amendment) before T07 ships.

## 11. Second pass summary (2026-05-22)

- **F-53 (E, §3.3) — Protected-modal focus-trap timing race during opacity transition.** Cross-ref a11y A-2; design-system §3.1/§3.2; RA-1. The ambiguity about whether focus trap engages on `modal.show()` or on `transitionend` creates a 100–200ms window during which a scripted attacker (A2 malicious extension, A3 coerced rep with helper) can race the export interstitial, the reauth prompt, the passphrase prompt, the destructive_confirm, or the four_eyes_pending modals. **Materially narrows RA-1 compensating-control margin** — the consent-shape of the confirm is part of why single-signer was acceptable; if confirm can be raced by 200ms, the case for HG-1 leaning toward 4-eyes strengthens. Three testable invariants: M-53a (trap-on-mount, synchronous with `modal.show()`), M-53b (announce-on-open `ready` promise gates input acceptance), M-53c (underlying surface inert with `aria-hidden=true` + `inert` + scrim capturing keydown from t=0).
