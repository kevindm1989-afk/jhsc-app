## 1. System overview and trust boundaries

The worker-side JHSC app is a single-tenant, single-committee, worker-only PWA supporting the worker side of an Ontario Joint Health and Safety Committee (OHSA s.8–9). The adversary model is unusual and load-bearing: the **employer is treated as a credible, well-resourced adversary** alongside the conventional set of external attackers, insider threats, and platform compromises. The system must remain unreadable to:

- **A1 — Employer via Canadian or US legal process** (subpoena, production order, CLOUD Act / NSL / FISA 702 reaching Supabase Inc. or AWS).
- **A2 — Employer via a worker's employer-owned device** (MDM, screen-recording, key-logging, certificate interception on employer networks).
- **A3 — Co-opted or coerced insider rep** ("under duress" rep handing data to the employer in exchange for favour or under threat of OHSA s.50 reprisal).
- **A4 — Conventional external attacker** (phishing, session theft, dependency compromise, supply-chain).
- **A5 — Hosting provider compromise / rogue admin** (Supabase Inc. employee or compromised infrastructure).

The hosting decision (ADR-0001) is only defensible because the E2EE model (ADR-0003) holds. **Every STRIDE finding below is graded with reference to whether the E2EE boundary still confines plaintext to the user's browser.**

### Trust boundaries (architect's three, ratified, plus two surfaced here)

1. **B1 — Auth boundary (Supabase Auth perimeter).** Unauthenticated requests → authenticated principal. WebAuthn + short-TTL sessions. Outside B1: zero implicit trust. Inside B1: a named `auth.uid()` that RLS uses.
2. **B2 — E2EE boundary (client / server data plane).** Plaintext lives ONLY in the browser (libsodium buffers, Svelte component state, transient IndexedDB session-wrapping layer). Anything crossing toward the server MUST be ciphertext for C3/C4 data. Edge Functions are server-side and never see plaintext.
3. **B3 — Worker/employer boundary (the export function).** The single egress path off the worker side. PDF rendered in-browser by the worker co-chair, downloaded, delivered off-app. Logged in audit log.
4. **B4 — Browser session boundary (added here).** The architect's diagram conflates the IndexedDB-on-disk state with browser RAM. This threat model splits them: the contents of IndexedDB are at rest on a personal device and survive crash/reboot; the in-memory plaintext does not. Different threat surfaces (T2 employer-owned device hits IndexedDB; T6 device theft hits both). The implementer treats IndexedDB as a write-through ciphertext store with a session-scoped wrapping layer; raw plaintext is not allowed to persist.
5. **B5 — Backup/restore boundary (added here).** The nightly `pg_dump` crosses from the live Supabase project into an independent Canadian B2 bucket. The architect treats this as a ciphertext blob (correct), but the *dump encryption key* is its own escrow asset (paper + 1Password) — its compromise yields the same metadata the platform sees, plus the C1 audit log content. Worth its own trust boundary because the controls (key escrow procedure) are different.

## 2. Data flow diagrams (per priority component)

### 2.1 E2EE key handling (T07, ADR-0003)

```
[First-login enrollment]
  Browser: libsodium.crypto_box_keypair() -> {ident_pub, ident_priv}
  Browser: passkey-derived secret K_p = WebAuthn PRF / passkey-wrap (T05 + T07)
  Browser: ident_priv_wrapped_local = secretbox(ident_priv, K_p)
  Browser -> IndexedDB: store ident_priv_wrapped_local      [B4]
  Browser: prompt user to print recovery passphrase R; user types it back to confirm
  Browser: K_r = argon2id(R, salt)
  Browser: ident_priv_recovery_blob = secretbox(ident_priv, K_r)
  Browser -> POST /users/me/recovery_blob { ident_priv_recovery_blob, salt }   [B2, ciphertext]
  Server: stores in users.identity_privkey_recovery_blob (C2)

[Committee data-key generation — done once by first co-chair]
  Browser: libsodium.crypto_box_keypair() -> {ck_pub, ck_priv}
  Browser: wrap_for_self = crypto_box_seal(ck_priv, self.ident_pub)
  Browser -> POST /committee_key  { ck_pub, wrap_for_self }   [B2]
  Server: committee_key.public = ck_pub (C1); committee_key.wrapped[self] = wrap (C3)

[Add member (existing member M re-wraps for new member N)]
  M browser: unseal own wrapped ck_priv = crypto_box_seal_open(self.wrap, self.ident_priv)
  M browser: fetch N.ident_pub (server)
  M browser: wrap_for_N = crypto_box_seal(ck_priv, N.ident_pub)
  M browser -> POST /committee_key/wrap  { member_id: N, wrap: wrap_for_N }   [B2]

[Remove member X (rotation)]
  Remaining member R browser: takes per-committee advisory lock via Edge Function
  R browser: unseal own ck_priv (old key)
  R browser: generate new committee keypair {ck_pub', ck_priv'}
  R browser: for each remaining member Y including self: wrap_for_Y = crypto_box_seal(ck_priv', Y.ident_pub)
  R browser -> POST /committee_key/rotate { new_pub: ck_pub', new_wraps: [...] }   [B2, atomic]
  Server: in single transaction: (a) update committee_key.public; (b) replace all wraps with new wraps;
          (c) delete X's wrap; (d) audit log; (e) release lock.
  Server: NEW C3/C4 writes are sealed to ck_pub'. OLD ciphertext (sealed to ck_pub) remains; remaining
          members can still decrypt via their OLD wrap (kept under a `committee_key_history` row keyed
          on rotation_id) — X's old wrap is destroyed so X cannot.

[Submission of a concern body (sealed-box submission)]
  Browser: ct = crypto_box_seal({title, body, source_name?}, ck_pub)
  Browser -> POST /concerns { ct, metadata }   [B2]
  Server: row in concerns: ct columns + metadata; audit log row.

[Decrypt path]
  Browser: fetch self.wrap (current) and the ciphertext
  Browser: ck_priv = crypto_box_seal_open(self.wrap, self.ident_priv)
  Browser: plaintext = crypto_box_seal_open(ct, ck_pub, ck_priv)
  (For C4: an additional layer with a per-record key wrapped under ck_pub; a second decrypt step.)
```

### 2.2 Concern intake (T08, ADR-0007)

```
[Auth gate] Rep is authenticated; RLS evaluates is_active_member() = true.
[Form] Rep opens intake; anonymous toggle defaults ON. The source-name input is hidden behind the toggle.
[Client encrypt] Title + body + (optional) source_name encrypted with ck_pub via sealed box.
                 If source_name present, an extra per-record key wraps a C4 secretbox over source_name.
[POST] /concerns { title_ct, body_ct, source_name_ct?, hazard_class, severity, location_id, anon: bool }   [B2]
[Server] RLS denies INSERT unless is_active_member(); INSERT row; audit log {action='concern.create'}.
[List view] Members SELECT (RLS pre-filtered); client decrypts blobs per row.
[Source reveal path] Member clicks "reveal source"; client prompts per-record passphrase (C4);
                     audit log {action='source.reveal'} appended BEFORE the plaintext is shown
                     (so even an aborted reveal is recorded).
```

### 2.3 Export to employer co-chair (T11 + T12)

```
[Auth + role] Acting principal MUST be worker_co_chair (RLS gates the read of finalized minutes).
[Field selection (whitelist)] Export builder reads from a hard-coded allowlist of column refs
  for the document type: {minutes.final: title, agenda_items, decisions, attendees(committee_role only)}
  {recommendations.export: title, body, sent_date, employer_response?}
  source_name is NOT in any allowlist; the allowlist is closed (no "..." spread).
[Client decrypt] Fetch ciphertext blobs; decrypt in-browser with ck_priv.
[Interstitial / sign-off]
  - List every included field by label (not value).
  - Banner: "This will leave the worker side. Continue?"
  - Co-chair confirms by re-asserting passkey (re-auth, not just click-through).
  - If practical, 4-eyes: a second worker member's passkey assertion required.
[Render] In-browser PDF (jsPDF or pdf-lib) -> Blob -> URL.createObjectURL -> <a download>.
  NO POST to any server-side renderer. NO Edge Function involvement in PDF bytes.
[Audit] BEFORE the blob is offered to the user, POST audit log
  {action='export.minutes' OR 'export.recommendation', target_id, recipient_role='employer_co_chair',
   field_set_hash, ts, approver_id?}
  on failure to log -> abort the export.
[Off-app delivery] User downloads PDF, delivers via email/paper. App is not the transport.
```

### 2.4 Reprisal log (T13)

```
[Write] Member -> form -> client encrypts body with ck_pub (sealed box) AND a per-record symmetric key
        wrapped to ck_pub (C4 second layer). Per-record passphrase is required to view, not to write.
[Read] Author / co-chair / certified member only. Client retrieves ciphertext, unwraps with ck_priv,
       then prompts per-record passphrase to unwrap the C4 inner layer.
[Sensitive-read notification] On unwrap (T11 surface), client POSTs audit log {action='reprisal.read'}
       BEFORE rendering plaintext. The "recent sensitive reads" view subscribes to this stream and
       surfaces to every active member.
[Delete] Two-member quorum via `pending_destructive_ops` table. Member A proposes; member B (distinct)
       approves; only then RLS permits DELETE. Audit log captures both approvers.
[Retention] Active matter + 7 years from last-activity; real delete on schedule; crypto-shred backstop
       when committee key rotates and the old wraps are destroyed.
```

### 2.5 Auth + session model (T05, ADR-0002)

```
[First-device enrollment]
  Co-chair generates TOTP invite slip (paper or email).
  User loads /enroll; enters TOTP; server validates; user binds first passkey via WebAuthn (platform or roaming).
  Server destroys TOTP secret atomically with passkey binding (single transaction).
[Subsequent device]
  Existing passkey-authenticated session approves the new authenticator (no TOTP re-issue).
[Session]
  Access token TTL <= 15 min. No long-lived refresh token. Re-auth on resume via passkey assertion.
[Revocation]
  Settings -> "Sessions": list of (device fingerprint, last-seen, current?), revoke per row or all.
  Revoke = server invalidates JWT family + posts audit log.
[Recovery]
  No passkey + no other enrolled device -> co-chair issues fresh TOTP invite (4-eyes if multiple co-chairs);
  identity-key rotation may follow (ADR-0002 + ADR-0003).
```

## 3. STRIDE analysis per component

Format per finding: **F-id | STRIDE | description | affected component | current mitigation (ADR/task ref) | residual risk | testable mitigation**.

### 3.1 Priority 1 — E2EE key handling (T07, ADR-0003)

#### F-01 | S | Forged committee-key wrap insertion by a co-opted member
A member with valid auth re-wraps `ck_priv` for a non-member identity public key (e.g., the attacker's own off-roster account, or a key the employer holds). Server only sees a wrap for member_id X; if X is not currently active, the wrap is dead, but if the attacker can also create a stub `users` row or flip an inactive member to active, the wrap becomes live.
- **Affected:** `committee_key.wrap` insert path (T07), `committee_membership` admin (T06).
- **Mitigation:** Co-chair-only INSERT on `committee_membership`; active flag managed by co-chair (RLS) (ADR-0004, T06). Invite flow requires an existing TOTP-issued account (T05).
- **Residual:** Medium — co-chair takeover (A3) can chain to this.
- **Testable mitigation (T07):** *"A wrap row whose `member_id` does not have `committee_membership.active = true` at the time of INSERT must fail RLS. Test: as member M, attempt to insert a wrap for user U where U is inactive; assert RLS denies. Repeat with U not in `committee_membership` at all; assert RLS denies. Repeat with U made active by co-chair after wrap insert; assert wrap was rejected on the insert and is not retroactively valid."*

#### F-02 | S | Spoofed identity public key on first enrollment
A user enrolls and posts an identity public key that is not the one paired with the locally-held private key (e.g., MITM on the enrollment POST, or a buggy client). Subsequent wraps to this `ident_pub` cannot be opened by the legitimate user; effectively a denial-of-access. If the attacker controls the corresponding private key, the attacker decrypts.
- **Affected:** `users.identity_pubkey` insert (T05/T07).
- **Mitigation:** TLS 1.3 + Supabase Auth JWT bound to the request. No second factor verifying public-key/private-key pairing.
- **Residual:** Medium — TLS protects transit but the server has no way to verify pubkey-privkey pairing.
- **Testable mitigation (T07):** *"At identity-key enrollment, client must perform a self-test challenge: server returns a nonce sealed to the just-posted ident_pub; client must unseal and return; if unseal fails, enrollment is rolled back and the row is deleted. Test: register an ident_pub whose privkey is on a different keypair; assert enrollment is rejected within one request-cycle and no `users.identity_pubkey` row persists."*

#### F-03 | T | IndexedDB identity-private-key blob tampered by malicious extension or browser plugin
A browser extension with broad permissions modifies `ident_priv_wrapped_local` in IndexedDB (replaces with attacker's wrap). On next session, user re-derives K_p via passkey; the local wrap fails to open OR opens to the attacker's chosen privkey.
- **Affected:** IndexedDB at-rest (B4).
- **Mitigation:** No mitigation in stack today beyond the OS+browser sandbox. Recovery-passphrase blob on server is unchanged.
- **Residual:** Medium-low — requires malicious extension installed on the user's PWA origin context.
- **Testable mitigation (T07):** *"On every session start, after unwrapping `ident_priv_wrapped_local`, the client must perform a self-test: decrypt a known sentinel (a fixed sealed box to ident_pub, stored on the server, immutable post-enrollment). If self-test fails, the client must (a) refuse to proceed, (b) prompt recovery-passphrase flow, (c) post a `client.identity_selftest_fail` audit-log entry. Test: corrupt the IndexedDB blob; assert client refuses to enter authenticated state and surfaces the recovery flow."*

#### F-04 | T | Rotation race: two members rotate simultaneously
Two members each take the (advisory) lock-free path of `committee_key/rotate`; the second overwrites the first's wraps, possibly with a key the first member has already used to encrypt new C3 writes against the stale public key.
- **Affected:** Rotation endpoint (T07).
- **Mitigation:** ADR-0003 invariant 6 — Edge Function takes a per-committee advisory lock.
- **Residual:** Low — invariant 6 covers it IF the test confirms it.
- **Testable mitigation (T07):** *"Concurrent rotation calls must serialize: spawn two simultaneous rotation requests for the same committee from different members; assert exactly one succeeds and the other returns HTTP 409 (lock held) within the lock TTL. Assert no `committee_key` row exists with a wrap set that mixes pre- and post-rotation wraps. Assert no transient state where `committee_key.public` is updated but wraps for remaining members are missing."*

#### F-05 | T | Removed member's wrap not fully purged
On removal of member X, X's `committee_key.wrap` row is supposed to be deleted. If the rotation is partial (writes succeed but DELETE fails), X retains decryption capability for OLD ciphertext until backup expiry.
- **Affected:** Rotation flow (T07).
- **Mitigation:** Invariant 6 atomicity.
- **Residual:** Medium without a test.
- **Testable mitigation (T07):** *"After rotation triggered by removal of member X, X's row in `committee_key` (current AND history) is absent. Test: remove X; query Postgres directly (admin connection) for any row in `committee_key*` with `member_id = X`; assert zero rows. Also assert X's session is invalidated within the same transaction (no race between rotate-complete and session-revoke)."*

#### F-06 | T | Forward-secrecy gap: backups still contain X's wrap
Even after rotation, the nightly `pg_dump` for the prior 35 days contains X's old wrap and so could decrypt OLD ciphertext IF the backup-encryption key AND X's identity key are both available. Backups are encrypted with an escrow key; X does not hold the escrow key; but a subpoena to Backblaze + an escrow-key compromise + X's cooperation chains the breach.
- **Affected:** Backup chain (T17, B5).
- **Mitigation:** Backup is itself a libsodium secretbox; the bucket holds ciphertext-of-ciphertext.
- **Residual:** Medium — documented as accepted in ADR-0012 ("by design, T1/T5").
- **Testable mitigation (T17):** *"`pg_dump` files in the bucket must not be readable without the escrow key. Test: pull a fresh dump from the bucket via the bucket's own admin credentials; attempt to open with libsodium secretbox using a random key; assert decrypt fails. Test the dump-key never appears in the same provider's metadata, tags, or object ACLs."*

#### F-07 | R | Re-wrap or rotation lacks attribution
Existing audit-log spec covers role changes and concern actions; an explicit "wrap added/deleted/rotated" event is not enumerated in the failure-mode analysis or PI inventory. Without it, a later forensic question ("who re-wrapped for member N? when?") cannot be answered.
- **Affected:** Audit log (T18) ∩ rotation flow (T07).
- **Mitigation:** None today — gap in architecture.
- **Residual:** Medium — repudiation surface for A3.
- **Testable mitigation (T07 + T18):** *"Every key-material mutation (committee_key insert, wrap insert, wrap delete, rotation) writes an audit-log row with action ∈ {'key.committee.create','key.wrap.add','key.wrap.delete','key.committee.rotate'} containing the actor, target member_id, rotation_id (where applicable), and prev-hash. Test: trigger each of the four operations and assert the audit-log row exists, chains correctly, and is visible to all active members."*  **FINDING:** architecture gap; surface to architect as ADR follow-up.

#### F-08 | I | Recovery-passphrase backup blob brute-force
`users.identity_privkey_recovery_blob` is on the server (C2 ciphertext). If an attacker exfiltrates it (A5) and the user chose a weak passphrase, offline brute-force is possible.
- **Affected:** Recovery-blob storage (T07).
- **Mitigation:** Argon2id KDF with high parameters; passphrase chosen client-side and required to be typed-back to confirm.
- **Residual:** Medium if Argon2 parameters are weak.
- **Testable mitigation (T07):** *"Argon2id parameters MUST be: ops=4, mem=2^19 (512MB) OR a documented per-user override floor. Test: capture an emitted recovery-blob; verify the embedded KDF parameters meet the floor. Bench a brute-force attempt of 10 candidate passphrases; assert wall-clock per attempt > 250ms on CI hardware."*

#### F-09 | I | Edge Function logs leak ciphertext provenance / shape
Edge Functions handle ciphertext payloads. If they log `req.body` for debugging, they leak who-uploaded-what and a payload size oracle, even if no plaintext.
- **Affected:** Edge Functions (T07, T11, T16).
- **Mitigation:** ADR-0010 PI scrubbing; ADR-0001 invariant 4/5.
- **Residual:** Low if linted; medium otherwise.
- **Testable mitigation (T02 + T07):** *"Edge Function code paths must not invoke `console.log(req)` or `console.log(req.body)` or any logger with the raw request object. Semgrep rule + CI test: deploy a function that does so on a side branch; assert CI rejects. At runtime: instrument tests fire C3 writes through an Edge Function; capture Supabase function logs; assert no ciphertext bytes appear."*

#### F-10 | I | Service worker caches plaintext responses for offline
SvelteKit service worker may auto-cache decoded responses. If decrypted plaintext ends up in `caches.open(...)`, T2 (employer-owned device) trivially reads it from the cache without re-auth.
- **Affected:** Service worker (T10).
- **Mitigation:** Architect notes "no plaintext caching server-side" but the client-side service-worker cache rules are not in the ADRs.
- **Residual:** Medium — likely real gap.
- **Testable mitigation (T10):** *"Service worker MUST NOT cache responses for any route under `/api/*` that returns decrypted content. The service worker's cache routing rules are an explicit allowlist of static assets only (HTML shell, CSS, JS bundles, fonts, OHSA reference doc library which is C0). Test: hit a C3-returning route, then enumerate Cache Storage; assert zero entries for the request URL. Also: after decrypt-in-browser, assert no `caches.put` is invoked with plaintext as Response body."*  **FINDING:** architecture gap; surface as a service-worker policy ADR addendum.

#### F-11 | I | Memory disclosure via DevTools / heap snapshot
A user with DevTools access (or a remote-controlled browser via Chrome Remote Debug) can heap-dump and find plaintext.
- **Affected:** Browser RAM (B4 in-memory).
- **Mitigation:** None achievable in a browser PWA.
- **Residual:** Accepted — out of scope (the threat model assumes the user's local machine is uncompromised; T2 handles employer-owned device via session hygiene; T6 handles theft).
- **Testable mitigation:** N/A — accepted limitation. Documented in "Open threats / accepted risks" (§7).

#### F-12 | D | Identity-key generation DoS via WASM exhaustion
Repeated forced key generation (e.g., a hostile re-onboarding loop) burns the user's CPU; also burns Edge Function quota if the recovery-blob upload is unauthenticated/replayable.
- **Affected:** Enrollment (T05 + T07).
- **Mitigation:** Enrollment requires a valid TOTP invite consumed once.
- **Residual:** Low.
- **Testable mitigation (T07):** *"Recovery-blob upload endpoint accepts at most one POST per user; subsequent POSTs return 409 unless preceded by an explicit `users.reset_recovery` row (co-chair-issued, audit-logged). Test: POST twice; assert second 409."*

#### F-13 | E | Member promotes self via direct row update
A non-co-chair member attempts to update `committee_membership.role` to include `worker_co_chair` directly, bypassing the invite flow.
- **Affected:** `committee_membership` (T06).
- **Mitigation:** ADR-0004 RLS — co-chair-only UPDATE.
- **Residual:** Low if RLS test exists.
- **Testable mitigation (T06):** *"As a non-co-chair member, attempting to UPDATE `committee_membership.role` for self or any other row is rejected by RLS. Test (positive): co-chair can update. Test (negative): worker_member cannot. Test (negative): certified_member cannot. Assert all three with explicit Postgres-level role assertions."*

#### F-14 | E | Co-chair → "DB superuser" privilege creep via Edge Function
An Edge Function created with `SECURITY DEFINER` and owned by an over-privileged role can be invoked from a co-chair session to read across RLS.
- **Affected:** Edge Functions (T07, T18, T16).
- **Mitigation:** Architect note that audit_log is written via `SECURITY DEFINER` definer — broader principle is implicit.
- **Residual:** Medium — easy to introduce regression.
- **Testable mitigation (T04 + T07):** *"All `SECURITY DEFINER` functions are owned by a `migration_role` that has explicit, minimal grants. Test: list every `SECURITY DEFINER` function via `pg_proc`; assert each is owned by `migration_role` and has a documented per-function justification in the migration file's comment. CI fails on any new `SECURITY DEFINER` whose comment header is missing."*

### 3.2 Priority 2 — Concern intake (T08, ADR-0007)

#### F-15 | S | Non-committee account submits via direct API call
An attacker with any Supabase Auth user (possible only via co-chair invite — but a removed-but-still-valid JWT, or a future bug) calls `POST /api/concerns` directly, bypassing the form.
- **Affected:** `/api/concerns` (T08).
- **Mitigation:** RLS denies INSERT unless `is_active_member()`.
- **Residual:** Low if RLS test exists.
- **Testable mitigation (T08):** *"`is_active_member() = false` users cannot INSERT into `concerns`. Test (positive): active member inserts; row appears. Test (negative): authenticated user with no `committee_membership` row attempts INSERT; assert RLS denies. Test (negative): authenticated user with `committee_membership.active = false` attempts INSERT; assert RLS denies. Test (negative): a removed member's still-valid JWT (T05) is invalidated within 60 seconds and the negative test then holds."*

#### F-16 | T | Post-submission edit of a concern without audit trail
A rep edits a concern body after submission (UPDATE); without explicit audit chain, the change is invisible later.
- **Affected:** `concerns` UPDATE (T08, T09).
- **Mitigation:** RLS allows UPDATE by active members; audit log spec covers writes but the per-field diff is not enumerated.
- **Residual:** Medium — audit captures the action but not the before/after, which matters for legal-relevance.
- **Testable mitigation (T08 + T09 + T18):** *"Every UPDATE on `concerns` writes an audit-log row with action='concern.update' AND `prev_field_hashes` (a JSON of `{field: sha256(prev_ciphertext)}`) so a later forensic query can detect that a body was changed without revealing plaintext. Test: update title; assert audit-log row has hash of prior `title_ct` and differs from the post-update value's hash."*

#### F-17 | R | Anonymous submission masks rep accountability if reps are coerced
A coerced rep submits a fake concern claiming a fictitious source to surface internal data for the employer. The audit log captures the submitting rep, so non-repudiation of the *rep* is preserved.
- **Affected:** `audit_log` (T18) + concern submission.
- **Mitigation:** Audit log always captures actor; "anonymous" applies to the source field, not the submitter.
- **Residual:** Low.
- **Testable mitigation (T08):** *"Every `concern.create` audit-log row has a non-null `actor_id` corresponding to the authenticated user, REGARDLESS of the anonymous-source toggle state. Test: submit with anonymous=true and anonymous=false; assert both audit rows carry the submitter's actor_id."*

#### F-18 | I | source_name leaks via list view or filter side-channel
A list view that filters by `source_name` content (e.g., "concerns for worker X") needs to read C4 plaintext server-side OR cause every row to be decrypted client-side. Either way, a careless implementation may include `source_name` in a default list payload to all members.
- **Affected:** `/api/concerns` list (T08).
- **Mitigation:** ADR-0007: name is E2EE and visible to all members on reveal. But the *default list payload* shape is not specified.
- **Residual:** Medium — implementation hazard.
- **Testable mitigation (T08):** *"Default list payload for `/api/concerns` does NOT include `source_name_ct` unless the requesting client explicitly opts in via a query flag `include=source_name` AND the user has already entered the per-record passphrase for that record (server tracks an ephemeral unlock token bound to the audit-log row). Test: GET list; assert no `source_name_ct` keys in response. Then explicitly request reveal; audit log gains 'source.reveal' BEFORE the ciphertext is returned; only then `source_name_ct` is in the payload."*  **FINDING:** strengthen list-payload contract.

#### F-19 | I | source_name leaks via export field whitelist gap (LAUNCH BLOCKER)
An export rendering accidentally includes `source_name_ct` because the field selection uses blacklist semantics ("everything except X") rather than a closed allowlist.
- **Affected:** Export builder (T11, T12).
- **Mitigation:** Plan §3.2: "the *only* artifacts ... do not include source identities".
- **Residual:** HIGH if not structurally enforced (this is the T3 launch blocker).
- **Testable mitigation (T11 + T12):** *"Export-builder field selection is a closed allowlist; the allowlist for every document type is a const exported from a single module `src/lib/export/allowlist.ts`; an ESLint rule forbids spread-into-export-payload outside that module. Test: snapshot test asserts the allowlist for `minutes.final` and `recommendations.export` equals an exact, hard-coded JSON array. Test: render a PDF for a concern carrying a source_name; assert the rendered text does NOT contain the source_name string anywhere (extract PDF text, grep). Test: every C4 field name is absent from every export allowlist."*

#### F-20 | D | Concern-creation flood by a single rep
A co-opted rep spams thousands of concerns to (a) bury real ones in noise, (b) exhaust Supabase quota.
- **Affected:** `/api/concerns` (T08).
- **Mitigation:** Plan §7 "Rate limiting on auth, concern submission, and export endpoints."
- **Residual:** Low if implemented.
- **Testable mitigation (T08):** *"Rate limit on `POST /api/concerns`: <= 20 per user per hour, <= 200 per user per 24h. Test: programmatic 25 POSTs within an hour from one user; assert 21st returns 429 with no row written. Test: rate-limit response carries no PI."*

#### F-21 | E | Regular member triggers certified-only action via API
A regular `worker_member` calls a `work_refusal` or `s51_evidence` endpoint that should be `certified_member`-only.
- **Affected:** RLS on `work_refusal`, `s51_evidence` (T14).
- **Mitigation:** RLS outline says `Certified member OR co-chair`.
- **Residual:** Low if test exists.
- **Testable mitigation (T14):** *"`worker_member` who is not `certified_member` cannot SELECT, INSERT, or UPDATE on `work_refusal` or `s51_evidence`. Test (positive): certified_member can. Test (negative): worker_member cannot. Test (negative): inactive certified_member cannot."*

### 3.3 Priority 3 — Export to employer co-chair (T11 + T12)

#### F-22 | S | Export triggered without co-chair role
An active member who is not co-chair invokes the export client flow and produces a PDF; the act of producing the PDF leaks plaintext off the worker side even though the audit log catches it after the fact.
- **Affected:** Export client (T11, T12) + RLS (T04).
- **Mitigation:** RLS gates the SELECT of finalized minutes by co-chair; recommendations same.
- **Residual:** Low if RLS holds.
- **Testable mitigation (T11 + T12):** *"As an active non-co-chair member, the GET that fetches finalized-minutes ciphertext is denied by RLS. Test: assert HTTP 403/404 (whichever) and no ciphertext in response."*

#### F-23 | T | PDF tampered between in-browser render and download
A malicious browser extension intercepts the Blob and modifies bytes before download.
- **Affected:** Browser local context (B4 in-memory).
- **Mitigation:** Cannot fully mitigate; the act of downloading is local.
- **Residual:** Low — out of scope (same envelope as F-11).
- **Testable mitigation:** N/A — accepted; document in §7.

#### F-24 | R | Export action not audit-logged before delivery
The co-chair downloads a PDF but the audit-log POST is fire-and-forget; on failure, the PDF reaches the user but there's no record.
- **Affected:** Export flow (T11 + T12 + T18).
- **Mitigation:** None specified — gap.
- **Residual:** Medium.
- **Testable mitigation (T11 + T12):** *"Audit-log POST for the export action MUST succeed BEFORE the Blob URL is created. Test: mock the audit-log endpoint to fail; assert the user receives an error and NO Blob URL is created and NO `<a download>` element is added to the DOM. Test: the success path logs the row, hash-chains it, and only then offers the download."*

#### F-25 | I | Server-side PDF rendering accidentally introduced
A future PR adds an Edge Function or serverless route that renders the PDF "for convenience"; plaintext crosses B2.
- **Affected:** Export pipeline (T11 + T12).
- **Mitigation:** ADR-0001 invariant 5 (Edge Functions ciphertext only) — but a non-Edge route could appear.
- **Residual:** Medium without a structural test.
- **Testable mitigation (T11 + T12):** *"CI route inventory test: assert no server route under `/api/*` returns a PDF content-type. Semgrep rule: `application/pdf` content-type can only be set in client-side modules. Test: a deliberate violation in a PR fails CI."*

#### F-26 | I | source_name leak via export — covered F-19; LAUNCH BLOCKER cross-reference.

#### F-27 | I | Field-set-hash in audit log is not bound to what was actually rendered
The audit log captures `field_set_hash` but a buggy client could log the *intended* allowlist while rendering with a different one.
- **Affected:** Export audit-log path (T11/T12 + T18).
- **Mitigation:** Single allowlist source-of-truth (F-19).
- **Residual:** Low if structural.
- **Testable mitigation (T11 + T12):** *"The export client computes the allowlist hash from the same module reference used to render the PDF. Test: monkey-patch the rendering function to use a different allowlist; assert the hash mismatch triggers an export abort and a 'export.integrity_fail' audit-log entry."*

#### F-28 | D | Export endpoint flooded to fill audit log
Rapid spurious export attempts fill the audit log; not actual harm (audit log is C1) but operational noise.
- **Affected:** Export flow (T11 + T12).
- **Mitigation:** Rate limit on exports (plan §7).
- **Residual:** Low.
- **Testable mitigation (T11 + T12):** *"Rate limit: <= 10 export attempts per co-chair per hour; 11th returns 429 with no audit-log noise written (rate limit response itself is logged once with a `rate_limit` action)."*

#### F-29 | E | "4-eyes" on export is described as optional ("if practical")
Brief says "Verify exports require the worker co-chair's explicit sign-off (4-eyes if practical)". A weak "if practical" is a finding for v1.
- **Affected:** Export sign-off flow (T11 + T12).
- **Mitigation:** Architect did not lock 4-eyes; passkey re-auth at minimum.
- **Residual:** Medium — open architectural question.
- **Testable mitigation:** *Two posture options for human gate:* (A) v1 ships with passkey re-auth only and a hard requirement that the audit log captures `approver_id = actor_id` (single signer); 4-eyes deferred to v2 with rationale. (B) v1 ships with 4-eyes (any other active member's passkey assertion) and the audit log captures distinct `approver_id`. **Test for (B):** *"Same actor cannot be approver; test: actor approves own export, assert HTTP 403."*  **FINDING:** human-gate decision needed (see §7).

### 3.4 Priority 4 — Reprisal log (T13)

#### F-30 | S | Reprisal-log read by removed-but-cached-session member
A member removed from the committee but with an unexpired JWT reads a reprisal-log row before session revocation propagates.
- **Affected:** `reprisal_log` SELECT (T13) + session revocation (T05).
- **Mitigation:** 15-min access token TTL bounds the window; revocation invalidates JWT family.
- **Residual:** Medium — a 15-min window of read access is non-trivial for C4.
- **Testable mitigation (T13 + T05):** *"On `committee_membership.active` flip to false, the user's outstanding JWTs MUST be invalidated immediately (server-side jti revocation, not just rotation). Test: member M has a valid JWT, co-chair removes M; within 5 seconds, M's GET on `reprisal_log` returns 401, not 200."*

#### F-31 | T | Reprisal-log row edited to neuter the report
A co-opted rep edits a reprisal-log entry to remove employer-identifying detail.
- **Affected:** `reprisal_log` UPDATE (T13).
- **Mitigation:** RLS allows author OR co-chair UPDATE; audit log captures change.
- **Residual:** Medium — author + co-chair collusion possible.
- **Testable mitigation (T13 + T18):** *"Every UPDATE on `reprisal_log` records `prev_field_hashes` in audit log AND a `sensitive.update` row visible in the same 'recent sensitive activity' feed as reads. Test: update a row; assert all active members see the activity entry within 60 seconds."*

#### F-32 | R | Reprisal-log destructive op without quorum
A co-chair deletes a reprisal-log row without the second approver.
- **Affected:** `reprisal_log` DELETE (T13).
- **Mitigation:** ADR system design — `pending_destructive_ops` with two distinct approvers; RLS gates DELETE.
- **Residual:** Low if test exists.
- **Testable mitigation (T13):** *"DELETE on `reprisal_log` requires a row in `pending_destructive_ops` with two distinct `approver_id`s. Test: as co-chair, attempt DELETE directly; assert RLS denies. Propose; same co-chair approves own proposal; assert RLS denies. Different active member approves; assert DELETE succeeds and TWO audit-log rows exist (`proposal`, `approval`) chained correctly."*

#### F-33 | I | Reprisal-log read not surfaced as a sensitive event
A read by an authorized member does not appear in the "recent sensitive reads" feed because the audit-log write is bypassed (e.g., client falsely posts no read event).
- **Affected:** Sensitive-read pipeline (T13 + T18).
- **Mitigation:** ADR system design says reads visible to all members; failure mode T11 test covers it.
- **Residual:** Medium without server-side enforcement.
- **Testable mitigation (T13 + T18):** *"Reading a `reprisal_log` row is server-instrumented: an Edge Function (or RLS-via-view trigger) writes the `reprisal.read` audit-log row on every successful SELECT, not just on client-initiated logging. Test: bypass the client and read via direct SQL with a member's JWT; assert audit log gains a row regardless."*  **FINDING:** require server-side enforcement of the read-audit, not client-cooperative logging.

#### F-34 | I | Per-record passphrase weakness
The per-record passphrase for C4 is freely chosen by the recording member; weak passphrase + breach of `committee_key.wrap` yields plaintext.
- **Affected:** C4 per-record key wrapping (T13).
- **Mitigation:** Per-record key is symmetric and wrapped to the committee public key; the passphrase only unlocks a *client-side* prompt before render (not the cryptographic gate).
- **Residual:** Low if the architecture is "passphrase is UX gate, not crypto gate." Medium if implementer treats it as the crypto gate.
- **Testable mitigation (T13):** *"The per-record passphrase is a UX confirmation only; the cryptographic gate is `ck_priv` + the per-record key wrapped to `ck_pub`. Test: an attacker who has `ck_priv` (e.g., a current member) can decrypt the per-record body WITHOUT the per-record passphrase. The passphrase requirement is a friction layer that DOES log the read attempt (F-33) even when wrong. Test: enter wrong passphrase three times; assert plaintext NOT shown AND a `sensitive.access_attempt` row written."*  **FINDING:** clarify semantics in T13 spec.

#### F-35 | D | Reprisal-log feed flooded by spurious entries
A rep submits many fake entries to overwhelm the committee's attention.
- **Affected:** `reprisal_log` INSERT (T13).
- **Mitigation:** Rate limit + audit log + 4-eyes on DELETE (deletion gated, but submission isn't).
- **Residual:** Low — friction is the right level here; reprisal needs to be writeable freely.
- **Testable mitigation (T13):** *"Rate limit on `reprisal_log` INSERT: <= 10 per actor per hour. Test: 11th returns 429."*

#### F-36 | E | Co-chair-only delete bypassed via direct UPDATE to set tombstone
A co-chair sets `status='deleted'` directly when DELETE is gated; if any retention job or list view filters by `status != 'deleted'`, the effect is a silent delete with no 4-eyes.
- **Affected:** `reprisal_log` UPDATE (T13).
- **Mitigation:** None explicit.
- **Residual:** Medium — pattern is "if a delete path exists outside DELETE, gate it the same way."
- **Testable mitigation (T13):** *"Changing `reprisal_log.status` to any value indicating removal (e.g., 'deleted','redacted','tombstoned') requires the same 4-eyes flow as DELETE. RLS enforces via a CHECK constraint reading `pending_destructive_ops`. Test: as co-chair, set status='deleted' directly; assert RLS denies until quorum exists."*  **FINDING:** add to T13 acceptance.

### 3.5 Priority 5 — Auth + session model (T05, ADR-0002)

#### F-37 | S | Passkey origin not bound to RP-ID
A passkey enrolled for a different origin or with a misconfigured Relying Party ID can be exercised against the production origin.
- **Affected:** WebAuthn config (T05).
- **Mitigation:** ADR-0002 + T7 test.
- **Residual:** Low if T7 test runs.
- **Testable mitigation (T05):** *"WebAuthn RP-ID is exactly the production origin's eTLD+1 (e.g., `jhsc.example.ca`). Test: attempt to use the passkey from `evil.example.ca` (subdomain); WebAuthn navigator API rejects with NotAllowedError. Repeat for `jhsc-example.com`; reject."*

#### F-38 | S | TOTP invite reused or guessed
A TOTP-bootstrap code emailed to a user is intercepted (employer-MITM on email) or brute-forced before consumption.
- **Affected:** TOTP bootstrap (T05).
- **Mitigation:** ADR-0002 — single-use; consumed atomically with passkey enrollment.
- **Residual:** Medium — interception of the email is plausible (A2 reach into worker's personal mailbox is harder but possible).
- **Testable mitigation (T05):** *"TOTP bootstrap: 6-digit code valid for <= 15 minutes, single-use, destroyed on successful passkey enrollment. Test: consume; reuse; assert second attempt 410 Gone. Test: 100 random guesses against a single user fall to rate-limit (<= 5 attempts / 15 min / user) and lock the invite, requiring co-chair re-issue."*

#### F-39 | T | Session token replay after revoke
A captured JWT continues to work after the user "revoked all sessions" if revocation is only client-side (cookie clear).
- **Affected:** Session revocation (T05, T19).
- **Mitigation:** Server-side jti tracking.
- **Residual:** Low if implemented.
- **Testable mitigation (T05 + T19):** *"Server tracks revoked jtis (or current-jti-per-session-id) for the JWT's remaining lifetime. Test: capture a valid JWT; call revoke-all; within 5 seconds, replaying the JWT returns 401 within one round trip. Hold the JWT for 10 minutes; assert it remains 401."*

#### F-40 | R | Account enumeration via differential error
Auth endpoint returns "unknown user" vs "wrong credential"; an attacker enumerates committee members.
- **Affected:** Auth endpoints (T05).
- **Mitigation:** ADR T8 test + plan §13.
- **Residual:** Low if T8 test passes.
- **Testable mitigation (T05):** *"Auth endpoint responses for unknown vs known-with-wrong-credential are byte-identical (status, headers, body) AND timing-equivalent within 50ms tolerance. Test as failure-mode T8 spec."*

#### F-41 | I | Recovery-passphrase printed page coercion
A worker is coerced to surrender the printed recovery passphrase + the device; attacker decrypts identity privkey, unwraps committee key, reads all C3/C4 the user could see.
- **Affected:** Recovery-passphrase pattern (T07).
- **Mitigation:** None in v1 (duress mode is v2 per plan §14, ADR §13 item 5).
- **Residual:** Accepted; documented.
- **Testable mitigation:** N/A — accepted; tracked in §7.

#### F-42 | D | Auth endpoint flood
Repeated WebAuthn assertions or TOTP attempts exhaust quota.
- **Affected:** Auth (T05).
- **Mitigation:** Rate limit.
- **Residual:** Low.
- **Testable mitigation (T05):** *"Rate limits: <= 10 WebAuthn auth attempts per user per minute; <= 5 TOTP attempts per invite. Test: exceed each; assert 429 with no user-enumerating body."*

#### F-43 | E | TOTP secret retained after passkey set
A bug leaves the TOTP secret usable after the first passkey binds, giving an interception attacker a permanent bypass.
- **Affected:** TOTP destruction (T05).
- **Mitigation:** ADR-0002 — destroyed in the same transaction.
- **Residual:** Low if test exists.
- **Testable mitigation (T05):** *"Single SQL transaction commits both (a) passkey credential row insert AND (b) TOTP secret row delete (or null + flag). Test: bind first passkey; query for any TOTP row associated with user; assert zero rows AND `users.totp_destroyed_at IS NOT NULL`. Test: attempt TOTP login after passkey set; assert 401."*

### 3.6 Lower priority — Inspection sync + offline cache (T10)

#### F-44 | T | IndexedDB inspection queue tampered to inject false photos
A malicious extension or local attacker writes a forged inspection row into the queue; on sync, the row is uploaded as if authored by the user.
- **Affected:** Inspection offline queue (T10).
- **Mitigation:** Photos and notes encrypted with per-record key wrapped to committee key BEFORE queueing; the wrap is signed implicitly by the wrapping public key. An injected row encrypted to ck_pub by an attacker still arrives as a row authored by the authenticated session.
- **Residual:** Medium — the attacker can write attacker-controlled plaintext that becomes "the inspector's note."
- **Testable mitigation (T10):** *"Each queued item carries an HMAC over (ciphertext, sequence_number, user_id) keyed by a device-bound session key derived from the passkey. On sync, server verifies HMAC matches the JWT's user_id. Test: tamper a queue entry; assert sync rejects. Test: drop a queue entry from another user (copy from device B to device A); assert sync rejects."*  **FINDING:** offline-queue integrity is not in the ADR; surface.

#### F-45 | I | IndexedDB holds plaintext between sessions
If the client does not wrap IndexedDB-cached committee key with a session-only key, then between sessions an at-rest attacker (T2 employer device, T6 theft) reads the cache without re-auth.
- **Affected:** IndexedDB layer (B4, T10).
- **Mitigation:** Architect note "even local cache is ciphertext at rest in IndexedDB, modulo a session-key wrapping layer for quick read in current session" — needs concrete spec.
- **Residual:** Medium without explicit policy.
- **Testable mitigation (T10 + T19):** *"On session end (timeout, lock, tab close), the session-key in memory is destroyed; IndexedDB then holds only the passkey-wrapped privkey + sealed ciphertext blobs. Test: end session; inspect IndexedDB; assert every entry is one of: (a) ciphertext blob, (b) passkey-wrapped privkey, (c) public metadata (e.g., schema version, last-sync-id). Assert no plaintext field with content from C2/C3/C4 sources is present."*

#### F-46 | I | GPS metadata leak through opt-in poorly defaulted
GPS off by default per plan, but the EXIF on a photo may carry location even with the app's GPS toggle off.
- **Affected:** Photo capture (T10).
- **Mitigation:** Plan §8 doesn't explicit-strip EXIF.
- **Residual:** Medium — finding.
- **Testable mitigation (T10):** *"Client strips ALL EXIF metadata from photos BEFORE encryption. Test: feed a photo with known GPS EXIF; capture the bytes pre-encryption; assert no EXIF block present. Test: extract from upload bytes after encryption (decrypt with test key); assert no EXIF."*  **FINDING:** EXIF strip is not in T10 acceptance — add.

#### F-47 | D | Sync queue grows unbounded offline
A device offline for a month accumulates a queue large enough to OOM the browser on sync.
- **Affected:** Offline queue (T10).
- **Mitigation:** None explicit.
- **Residual:** Low.
- **Testable mitigation (T10):** *"Queue capped at 500 items; oldest-first sync. Test: insert 600 items; assert oldest 100 either dropped (with audit log) or that the cap blocks new inserts with a user-visible warning."*

### 3.7 Lower priority — Backup encryption chain (T17)

### 3.8 Lower priority — Audit-log integrity + retention (T18 + T16)

#### F-50 | T | Hash chain tampering not detected fast enough
A platform admin (A5) modifies an audit-log row; daily integrity job may run hours later.

- **Mitigation:** Daily check (ADR system design).
- **Residual:** Medium — 24h detection window.

- **Affected:** Retention job (T16).

## 5. Breach-notification trigger map (PIPEDA s.10.1)

PIPEDA s.10.1: notify OPC + affected individuals when a breach creates **real risk of significant harm** (RROSH). Record-keeping for all breaches regardless of severity: **24 months**. See `playbooks/incident-response.md` for response steps.

| Data class | Example breach scenario | RROSH? | Notify OPC? | Notify individuals? | Time-to-notify | Response anchor |
|---|---|---|---|---|---|---|
| C0 Public | Document-library content exfiltrated | No | No | No | n/a | n/a |
| C1 Operational metadata | Audit log dumped (subpoena or A5 leak): timestamps, action verbs, user IDs (UUIDs), prev_hash chain. Reveals who-acted-when but not content. | **Sometimes** — for a JHSC, the fact that a worker U accessed reprisal_log at time T can itself be reprisal-precipitating evidence. | Yes (precautionary) | Yes — every active member | "As soon as feasible" — within 5 business days of confirmation | playbooks/incident-response.md §2 (Notification) + §4 (Coercion/duress) |
| C2 Worker-side PI | `users.display_name` + `off_employer_contact` leaked from DB at rest (provider admin or backup-key compromise) | **Yes** — combined with the fact that these are JHSC member contacts in an OHSA s.50 context, RROSH is met | Yes | Yes — affected members | ASAP, target <72h | playbooks/incident-response.md §2 |
| C3 Sensitive worker content | (Practically unreachable) — would require E2EE break OR committee-key compromise + ciphertext exfil | **Yes** — concerns, hazards, draft minutes, inspections, recommendations | Yes | Yes — affected members AND originating workers (if identifiable) | ASAP, target <48h | playbooks/incident-response.md §3 (E2EE failure) |
| C4 Highest sensitivity | (Practically unreachable per C3 rationale plus per-record key) — reprisal log, s.43 notes, s.51 evidence, source_name | **Yes — always** | Yes | Yes — including potentially-identified non-app workers if a source was named | ASAP, target <24h | playbooks/incident-response.md §3 + §4 |
| Auth credentials (passkeys, TOTP) | Passkey enrollment table exfiltrated (no password equivalents); TOTP secrets exfiltrated mid-enrollment | C2 effective — phishing-resistant by design, but TOTP misuse is possible if intercepted pre-passkey | Yes if any TOTP secret confirmed exfiltrated and not yet consumed | Yes if TOTP secrets implicated | ASAP, target <72h | playbooks/incident-response.md §5 (Auth) |

**PHIPA-adjacent note:** Plan §3.1 treats injury/exposure data (s.51 evidence; some reprisal entries about health-causing reprisal) as PHIPA-grade. We do not have a Health Information Custodian, so PHIPA does not technically apply — but the **60-day notification window** is held as the *outer* limit; we target faster per PIPEDA RROSH. If scope ever expands to a custodian relationship, PHIPA layer is triggered and ADR is required.

**Crypto-shred as containment:** For C3/C4, the primary containment for "ciphertext exfiltrated + we still hold the key" is **key rotation** (destroy old wraps, rotate committee key); for "ciphertext exfiltrated + key compromised", containment is "the data is irreversibly disclosed" — recovery is legal + member notification. The incident-responder playbook owns the decision tree.

## 6. Verification of the seven E2EE invariants (ADR-0003)

Each invariant is restated, mapped to a test, and (where applicable) strengthened.

### Invariant 1 — Server never sees a private key in the clear
**Restated:** Identity privates exist client-side or as ciphertext-on-server. Committee data privates exist only as wrapped blobs.
**Testable as written?** Mostly. Strengthen to include negative test against telemetry channel.
**Strengthened mitigation:** *"For every Edge Function and every Supabase function-log entry produced during the test suite, assert no byte sequence matches the format of an unsealed libsodium private key (32-byte high-entropy block in a context labeled `priv`/`secret`/`key`). Active assertion: T07 fixture inserts a known-canary 32-byte 'private-key-shape' value via DevTools into client state, triggers a flow; canary must never appear in any captured server-side log or Sentry payload."*  Tests: T07-test-1.

### Invariant 2 — No "admin recovery"
**Restated:** No server-side mechanism to recover a forgotten passphrase + lost device.
**Testable as written?** Yes, but the test is mostly an "absence" test (no endpoint exists).
**Strengthened mitigation:** *"Route inventory test (CI): no route under `/api/*` exists that accepts (user_id, request_admin_recovery) and returns plaintext identity material. No Edge Function accepts a passphrase-bypass token. Test: attempt to call hypothetical `/api/admin/recover-identity` and `/api/admin/decrypt-as`; assert 404. Manual: documentation explicitly states 'no admin recovery'."*  Tests: T07-test-2.

### Invariant 3 — No plaintext caching server-side, including in Edge Functions
**Testable as written?** Yes — combine with F-09.
**Strengthened mitigation:** as F-09; plus *"Semgrep rule bans `console.*(req)`, `console.*(req.body)`, `console.*(payload)` in Edge Function modules; runtime fixture posts a known-canary plaintext to an Edge Function; capture Supabase function logs; assert canary absent."*  Tests: T07-test-3.

### Invariant 4 — All encryption is libsodium primitives
**Testable as written?** Yes.
**Strengthened mitigation:** *"Semgrep rule: bans `crypto.subtle`, `node:crypto`, `bcrypt`, `aes-*` imports outside `src/lib/crypto/`; allowlists `libsodium-wrappers` only. CI test: full repo `grep -r` for forbidden imports; assert clean. Test: drop a forbidden import in a side branch; assert CI fails."*  Tests: T07-test-4.

### Invariant 5 — Key material in URLs is forbidden
**Testable as written?** Yes.
**Strengthened mitigation:** *"Semgrep rule: bans URL query params named `key`, `secret`, `passphrase`, `priv`, `nonce`; bans `Location: ...?...key=` patterns; bans setting cookies with names matching key patterns. Runtime test: spider every route; assert no URL contains base64-shaped 32+ byte tokens in query strings."*  Tests: T07-test-5.

### Invariant 6 — Rotation is atomic
**Testable as written?** Yes (mapped F-04, F-05).
**Strengthened mitigation:** as F-04 and F-05 combined. **PLUS: handle backup-window forward-secrecy gap (F-06) as an *accepted* residual, documented in §7.** Tests: T07-test-6a (concurrent rotation), T07-test-6b (X's wrap deleted), T07-test-6c (session revoke same transaction).

### Invariant 7 — C4 records require a second decrypt step logged in the audit log
**Testable as written?** Yes, but original wording is loose ("require a second decrypt step logged"). Tighten:
**Strengthened mitigation:** *"Reading a C4 row writes a `sensitive.read` audit-log entry BEFORE the plaintext is returned to the client (server-side enforcement via SECURITY DEFINER view or Edge Function indirection). Test: SELECT a C4 row directly with a valid member JWT; assert the SELECT either (a) returns the ciphertext and the read is server-instrumented to log on retrieval, OR (b) is funneled through a view/function that requires a logging step. Assert the audit-log row appears within the same transaction as the read."*  **FINDING (cross-link F-33):** original wording invites client-cooperative logging. Tighten to require server-side enforcement.  Tests: T07-test-7 + T13-test-3.

### Additional invariant to add — Invariant 8 (proposed): Key-material mutation audit trail
**Proposed wording:** *"Every mutation of key material (committee_key insert/update; wrap insert/delete; rotation) emits an audit-log row with a defined enum action and hash-chains to the prior audit row."*  Justification: F-07 surfaced this gap. **HUMAN-GATE DECISION:** add Invariant 8 to ADR-0003 (lightweight ADR amendment) before T07 ships.
